# 1. 기획 배경: 콘텐츠는 많지만 선택은 어려운 상황

콘텐츠 플랫폼은 이미 일상 속에 깊게 들어와 있다. 유튜브, 넷플릭스, 쇼츠, 릴스처럼 우리는 언제든지 원하는 영상을 볼 수 있는 환경에 있다. 그러나 선택지가 많아졌다고 해서 콘텐츠 선택이 쉬워진 것은 아니다. 오히려 콘텐츠가 많아질수록 사용자는 “뭘 봐야 하지?”라는 선택 피로를 더 자주 경험한다.

이 문제는 특히 가볍게 콘텐츠를 소비하고 싶은 순간에 더 분명하게 드러난다. 예를 들어 밥을 먹으면서 볼 영상을 찾거나, 쉬는 시간에 짧고 재밌는 콘텐츠를 보고 싶거나, 이동 중에 부담 없이 볼 영상을 찾는 상황이 그렇다. 사용자는 분명 어떤 분위기의 콘텐츠를 원하는지 대략적으로는 알고 있지만, 그것을 정확한 검색어로 표현하기는 어렵다.

```
유튜브를 켰는데 막상 볼 영상이 없다.
밥을 먹으면서 가볍게 볼 영상을 찾고 싶은데 뭘 검색해야 할지 모르겠다.
짧고 재밌는 콘텐츠를 보고 싶은데 정확한 검색어가 떠오르지 않는다.
홈 화면 추천을 계속 넘기지만 지금 보고 싶은 영상이 잘 나오지 않는다.
```

이 상황의 핵심은 콘텐츠가 부족하다는 것이 아니다. 문제는 사용자가 자신이 지금 원하는 콘텐츠를 **검색 가능한 형태로 표현하기 어렵다**는 데 있다. 사용자는 “밥 먹을 때 보기 좋은 영상”, “짧고 가벼운 영상”, “지금 기분에 맞는 영상”, “부담 없이 틀어놓을 수 있는 영상” 같은 막연한 요구를 가지고 있다. 그러나 이런 요구는 검색창에 그대로 입력하기 애매하다.

물론 `밥친구`, `짧은 영상`, `먹방`, `가벼운 예능`처럼 몇 가지 키워드는 떠올릴 수 있다. 하지만 이 키워드들을 어떻게 조합해야 원하는 콘텐츠가 나올지 판단하기는 쉽지 않다. 결국 사용자는 홈 화면을 계속 넘기거나, 추천 결과를 새로고침하거나, 검색어를 바꿔가며 시간을 보내게 된다.

따라서 이 콘텐츠 기획은 다음 질문에서 출발한다.

```
사용자의 막연한 요구를 어떻게 검색 가능한 형태로 바꿀 수 있을까?
정확한 검색어 없이도 의미적으로 가까운 콘텐츠를 찾을 수 있을까?
추천 결과가 왜 나왔는지 사용자가 이해할 수 있게 만들 수 있을까?
```

이 콘텐츠의 목표는 사용자의 막연한 요구를 의미 기반 검색으로 처리하고, 추천 결과의 근거까지 설명할 수 있는 기술 흐름을 소개하는 것이다. 이를 위해 먼저 기존 검색과 추천 방식이 왜 이러한 상황을 충분히 해결하기 어려운지 간단히 살펴본다.

# 2. 기존 기술 및 방식의 한계

사용자가 콘텐츠를 찾는 방식은 크게 직접 검색과 플랫폼 추천으로 나눌 수 있다. 직접 검색은 명확한 검색어가 있을 때 효과적이지만, “밥 먹을 때 가볍게 볼 영상”처럼 요구가 막연한 상황에서는 검색어를 만드는 것 자체가 부담이 된다.

키워드 기반 탐색도 표현의 한계를 가진다. 사용자가 원하는 의미와 콘텐츠의 실제 의미가 비슷하더라도, 같은 단어가 포함되지 않으면 관련 콘텐츠를 놓칠 수 있다. 사용자의 요구는 상황, 분위기, 길이, 장르가 섞인 형태로 나타나는 경우가 많기 때문에 단순한 키워드 일치만으로는 충분하지 않다.

플랫폼 추천은 사용자가 직접 검색하지 않아도 콘텐츠를 제시한다는 점에서 편리하다. 그러나 기존 추천은 과거 시청 기록이나 클릭 기록의 영향을 많이 받는다. 평소에는 경제 영상을 자주 보던 사용자도 지금은 밥을 먹으면서 볼 가벼운 영상을 원할 수 있다. 이런 순간적인 요구 변화는 기존 추천 방식에서 사용자가 직접 조정하기 어렵다.

또한 추천 결과가 나오더라도 사용자는 그 결과가 왜 나왔는지 알기 어렵다. 최근 시청 기록 때문인지, 특정 키워드 때문인지, 유사한 사용자 행동 때문인지 확인하기 어렵기 때문에 추천 결과를 이해하거나 조정하기 어렵다.

| 기존 방식 | 한계 |
| --- | --- |
| 직접 검색 | 정확한 검색어가 필요함 |
| 키워드 기반 탐색 | 표현이 다르면 관련 콘텐츠를 놓칠 수 있음 |
| 플랫폼 추천 | 현재 요구를 즉시 반영하기 어려움 |
| 일반 추천 결과 | 추천 이유를 이해하기 어려움 |

결국 필요한 것은 더 많은 추천 결과를 보여주는 기능보다, 사용자의 막연한 요구를 검색 가능한 형태로 바꾸고, 그 결과가 어떤 기준으로 추천되었는지 설명할 수 있는 구조이다. 이 방향을 위해 본 콘텐츠에서는 **Vector Search**와 **Explainable Recommendation**을 핵심 기술로 다룬다.

# 3. 문제 해결을 위한 핵심 기술

앞서 살펴본 한계를 보완하기 위해 이번 콘텐츠에서는 두 가지 기술적 접근을 중심으로 다룬다. 첫 번째는 사용자의 막연한 요구와 콘텐츠 정보를 의미적으로 연결하는 **Vector Search**이고, 두 번째는 추천 결과가 왜 나왔는지 사용자가 이해할 수 있도록 근거를 제공하는 **Explainable Recommendation**이다.

| 핵심 기술 | 역할 |
| --- | --- |
| Vector Search | 사용자의 요구와 콘텐츠 정보를 의미적으로 비교해 추천 후보를 찾음 |
| Explainable Recommendation | 추천 후보가 사용자의 어떤 요구와 연결되었는지 설명함 |

프로젝트 구조는 다음 흐름을 기준으로 설계한다.

```
카드 선택
→ 사용자 요구 프로필 생성
→ Embedding
→ Vector Search
→ Top-K 추천 결과 추출
→ 추천 근거 생성
```

이 흐름에서 Vector Search는 사용자 요구 프로필과 콘텐츠 설명문을 같은 벡터 공간에서 비교해 추천 후보를 찾는 역할을 한다. Explainable Recommendation은 추천 후보가 사용자의 선택값, 콘텐츠 정보, 제외 조건과 어떻게 연결되는지 설명하는 역할을 한다. 두 기술은 사용자의 입력을 추천 결과와 설명으로 이어주는 하나의 추천 구조 안에서 함께 작동한다.

이번 콘텐츠의 핵심은 정확한 검색어가 없어도 사용자의 요구와 의미적으로 가까운 콘텐츠를 찾을 수 있는지, 그리고 그 결과가 왜 추천되었는지 사용자가 이해할 수 있게 만들 수 있는지에 있다. 먼저 사용자의 막연한 요구를 의미 기반으로 처리하는 Vector Search부터 살펴본다.

## 3-1. Vector Search

### Vector Search의 개념

Vector Search는 단어가 직접 일치하는지를 기준으로 검색하는 방식에서 벗어나, 입력과 콘텐츠의 **의미적 유사성**을 기준으로 가까운 항목을 찾는 검색 방식이다. 기존 Keyword Search가 “같은 단어가 포함되어 있는가?”를 중심으로 작동한다면, Vector Search는 “의미적으로 가까운가?”를 중심으로 작동한다.

예를 들어 사용자가 `공포`라고 검색했다고 하자. 키워드 검색에서는 제목이나 태그에 `공포`라는 단어가 포함된 콘텐츠가 우선적으로 검색된다. 그러나 사용자가 실제로 원하는 것은 `실화 괴담`, `미제 사건`, `스릴러 리뷰`, `범죄 심리 분석`, `무서운 분위기의 이야기 영상`처럼 표현은 다르지만 분위기나 의미가 가까운 콘텐츠일 수도 있다.

Vector Search는 이러한 상황에서 유용하다. 텍스트를 단순한 문자열로 비교하지 않고, 그 텍스트가 가진 의미를 숫자 벡터로 표현한 뒤 벡터 공간에서 가까운 항목을 찾기 때문이다. 이를 통해 검색 기준은 단어의 직접 일치에서 의미적 유사성으로 확장된다.

---

### Embedding의 역할

Vector Search를 수행하려면 먼저 텍스트를 숫자 벡터로 변환해야 한다. 이 과정을 **Embedding**이라고 한다. Embedding은 자연어 문장이나 콘텐츠 설명을 고차원 숫자 배열로 변환하는 과정이며, 의미가 비슷한 문장들은 벡터 공간에서 가까운 위치에 놓이도록 표현된다.

예를 들어 다음 문장들은 표현은 다르지만 의미적으로는 비슷하다.

```
밥을 먹으면서 가볍게 볼 수 있는 영상
혼밥할 때 보기 좋은 짧은 음식 리뷰
부담 없이 틀어놓기 좋은 먹방 콘텐츠
식사 중 보기 좋은 짧은 예능형 콘텐츠
```

이 문장들은 같은 단어를 반복하지 않아도 “식사 중 가볍게 볼 콘텐츠”라는 의미를 공유한다. Embedding을 사용하면 이러한 의미적 유사성을 숫자 벡터로 표현할 수 있다.

Vector Search의 기본 흐름은 다음과 같다.

```
사용자 요구 문장
→ Embedding
→ user_vector

콘텐츠 설명문
→ Embedding
→ content_vector

user_vector와 content_vector 비교
→ 의미적으로 가까운 콘텐츠 검색
```

본 프로젝트에서는 사용자가 선택한 키워드를 그대로 검색어로 사용하지 않는다. 긍정 선택과 부정 선택을 바탕으로 하나의 **사용자 요구 프로필**을 만들고, 콘텐츠의 제목, 설명, 태그, 요약을 하나의 **콘텐츠 설명문**으로 통합한다. 이후 두 텍스트를 같은 임베딩 방식으로 벡터화하고, 유사도 기준으로 Top-K 콘텐츠를 추출한다. 이때 부정 선택과 관련성이 높은 콘텐츠는 제외하거나 감점하는 방식으로 추천 결과를 조정할 수 있다.

### 콘텐츠 추천에서 Vector Search가 필요한 이유

콘텐츠 추천 상황에서 사용자의 요구는 항상 명확한 검색어로 표현되지 않는다. 사용자는 “밥 먹을 때 틀어놓기 좋은 영상”, “짧고 부담 없는 콘텐츠”, “가볍게 웃을 수 있는 영상”, “지금 기분에 맞는 편안한 콘텐츠”처럼 상황과 분위기가 섞인 요구를 가진다. 이런 요구는 하나의 명확한 단어로 정리되기 어렵다.

예를 들어 사용자의 요구가 다음과 같다고 가정할 수 있다.

```
밥을 먹으면서 가볍게 볼 수 있는 짧은 음식 리뷰 콘텐츠를 보고 싶다.
```

이 요구는 `혼밥할 때 보기 좋은 편의점 신상 리뷰`, `10분 안에 보는 음식 리뷰`, `식사 중 틀어놓기 좋은 가벼운 먹방` 같은 콘텐츠와 의미적으로 연결될 수 있다. 표현은 다르지만 “식사 중 가볍게 볼 수 있는 짧은 음식 콘텐츠”라는 의미에서는 가깝기 때문이다.

따라서 Vector Search는 정확한 검색어가 없는 상황에서 유용하다. 사용자의 요구를 문장형 질의로 구성하고 콘텐츠 설명문과 비교함으로써, 표현이 달라도 관련성이 높은 콘텐츠를 찾을 수 있다.

---

### Keyword Search와 Vector Search의 차이

Keyword Search와 Vector Search의 차이는 검색 기준에서 가장 분명하게 드러난다. Keyword Search는 단어의 직접 일치를 중심으로 하고, Vector Search는 의미적 유사성을 중심으로 한다.

| 구분 | Keyword Search | Vector Search |
| --- | --- | --- |
| 검색 기준 | 단어의 직접 일치 | 의미적 유사성 |
| 입력 방식 | 짧은 검색어 중심 | 문장, 설명, 맥락 중심 |
| 강점 | 정확한 단어 검색에 강함 | 표현이 달라도 관련 의미 검색 가능 |
| 한계 | 단어가 다르면 놓칠 수 있음 | 임베딩 품질과 데이터 구성에 영향을 받음 |
| 적합한 상황 | 정확한 키워드가 있을 때 | 요구가 막연하거나 표현이 다양할 때 |

이 비교에서 중요한 점은 두 방식의 적합한 상황이 다르다는 것이다. 정확한 단어 검색이 필요한 경우에는 Keyword Search가 효과적이다. 그러나 본 프로젝트처럼 사용자의 요구가 막연하고 표현이 다양하게 나타나는 추천 상황에서는 Vector Search가 더 적합한 기반이 된다.

---

### Vector Search의 한계와 설계 포인트

Vector Search가 모든 문제를 자동으로 해결하는 것은 아니다. 검색 품질은 임베딩 모델뿐 아니라, 사용자 요구와 콘텐츠 정보를 어떤 텍스트로 구성하느냐에 크게 영향을 받는다.

예를 들어 콘텐츠 설명이 너무 짧거나 모호하면 콘텐츠 벡터도 실제 의미를 충분히 담지 못한다.

```
나쁜 콘텐츠 설명:
오늘 먹어봤습니다.

좋은 콘텐츠 설명:
편의점 신상 라면을 먹어보는 짧은 음식 리뷰 영상입니다.
혼밥할 때 보기 좋고, 식사 중 가볍게 볼 수 있는 먹방 콘텐츠입니다.
```

사용자 요구도 마찬가지이다.

```
나쁜 사용자 요구:
먹방, 짧은 영상, 밥친구

좋은 사용자 요구:
밥을 먹으면서 가볍게 볼 수 있는 짧은 먹방 또는 음식 리뷰 콘텐츠를 선호함.
```

따라서 본 프로젝트의 핵심은 Vector Search 적용 자체에 머물지 않는다. 사용자의 선택값을 벡터화하기 좋은 문장형 요구 프로필로 바꾸고, 콘텐츠 정보 역시 검색 가능한 설명문으로 정리해야 한다. 이 설계가 제대로 이루어져야 사용자 요구와 콘텐츠 정보가 같은 벡터 공간에서 의미 있게 비교될 수 있다.

---

### Vector Search의 역할 정리

정리하면 Vector Search는 다음 역할을 한다.

```
1. 사용자의 막연한 요구를 문장형 검색 질의로 변환한다.
2. 사용자 요구와 콘텐츠 설명문을 같은 벡터 공간에서 비교한다.
3. 표현이 달라도 의미적으로 가까운 콘텐츠를 찾는다.
4. Top-K 추천 결과를 추출하는 기반이 된다.
5. 설명 가능한 추천이 작동할 수 있는 의미적 연결 구조를 제공한다.
```

의미적으로 가까운 콘텐츠를 찾는 것만으로 추천 흐름이 완성되는 것은 아니다. 사용자는 결과가 왜 나왔는지도 알고 싶어 한다. 특히 요구가 막연할수록 추천 결과가 어떤 기준으로 선택되었는지 설명하는 과정이 중요해진다. 이 지점에서 필요한 기술이 **Explainable Recommendation**이다.

---

## 3-2. Explainable Recommendation

### Explainable Recommendation의 개념

Explainable Recommendation은 추천 결과와 함께 그 결과가 나온 이유를 사용자에게 설명하는 추천 방식이다. 일반적인 추천 시스템은 콘텐츠를 제시하지만, 사용자는 그 결과가 최근 시청 기록 때문인지, 특정 키워드 때문인지, 현재 선택한 조건 때문인지 알기 어렵다.

본 프로젝트에서 설명 가능한 추천은 단순히 “이 영상을 추천합니다”라고 말하는 방식이 아니다. 사용자의 선택값, 콘텐츠 정보, 제외 조건을 연결해 다음 질문에 답하는 구조이다.

```
왜 이 콘텐츠가 추천되었는가?
사용자의 어떤 선택이 반영되었는가?
콘텐츠의 어떤 정보가 사용자 요구와 연결되었는가?
사용자가 피하고 싶은 조건은 얼마나 반영되었는가?
```

즉, Explainable Recommendation은 추천 결과를 사용자가 이해하고 조정할 수 있게 만드는 구조적 요소이다. 추천 근거가 함께 제공되면 사용자는 자신의 요구가 제대로 반영되었는지, 원하지 않는 요소가 포함되지는 않았는지 판단할 수 있다.

---

### 추천 근거 생성 구조

Explainable Recommendation은 추천 결과에 단순히 설명 문장을 붙이는 방식으로 끝나지 않는다. 추천 과정에서 사용된 사용자 요구, 콘텐츠 정보, 의미적 유사성을 다시 정리해 사용자가 이해할 수 있는 문장으로 재구성하는 과정이다.

기본 흐름은 다음과 같다.

```
사용자 요구 확인
→ 콘텐츠 정보에서 관련 요소 탐색
→ Vector Search 결과 확인
→ 사용자 요구와 콘텐츠 정보의 연결 관계 정리
→ 추천 근거 문장 생성
```

추천 근거는 크게 세 가지 기준으로 구성한다.

| 기준 | 설명 |
| --- | --- |
| 긍정 선택과의 연결 | 사용자가 선택한 키워드와 콘텐츠 태그·설명·요약이 어떻게 연결되는지 확인 |
| 의미적 유사성 | 사용자 요구 프로필과 콘텐츠 설명문의 벡터 유사도가 높은지 확인 |
| 부정 조건 반영 | 사용자가 피하고 싶은 조건과 관련성이 낮은지 확인 |

예를 들어 사용자의 요구와 콘텐츠 정보가 다음과 같다고 가정할 수 있다.

```
사용자 요구:
밥 먹을 때 가볍게 볼 수 있는 짧은 콘텐츠

콘텐츠 정보:
혼밥, 음식 리뷰, 10분 내외 영상

연결 근거:
식사 중 부담 없이 볼 수 있는 짧은 음식 콘텐츠
```

이 경우 추천 설명은 다음과 같이 구성할 수 있다.

```
이 콘텐츠는 사용자가 선택한 ‘먹방’, ‘짧은 영상’, ‘밥친구’ 조건과 관련성이 높습니다.
콘텐츠 설명에 ‘혼밥’, ‘음식 리뷰’, ‘10분 내외 영상’ 정보가 포함되어 있어
식사 중 부담 없이 볼 수 있는 콘텐츠로 추천되었습니다.
또한 사용자가 피하고 싶다고 선택한 ‘공포’, ‘긴 다큐’와 관련된 요소가 낮아
현재 선택 조건에 적합한 콘텐츠로 판단되었습니다.
```

이 설명은 사용자의 선택값, 콘텐츠 정보, 제외 조건을 함께 보여준다. 따라서 추천 결과가 임의로 제시된 것이 아니라, 사용자의 입력과 콘텐츠 데이터에 기반했다는 점이 드러난다.

---

### Vector Search와 Explainable Recommendation의 관계

Vector Search는 의미적으로 가까운 콘텐츠를 찾는 데 강점이 있다. 그러나 “사용자 벡터와 콘텐츠 벡터의 유사도가 높아 추천되었습니다”라는 설명만으로는 충분하지 않다. 사용자는 벡터 유사도 자체보다, 자신의 요구와 콘텐츠 정보가 어떻게 연결되었는지를 알고 싶어 한다.

따라서 Vector Search 결과는 사용자가 이해할 수 있는 언어로 다시 변환되어야 한다. 두 기술의 관계는 다음과 같이 정리할 수 있다.

```
Vector Search:
사용자 요구와 콘텐츠 설명문이 의미적으로 가까운지 계산한다.

Explainable Recommendation:
그 의미적 가까움이 어떤 사용자 선택과 어떤 콘텐츠 정보 때문에 발생했는지 설명한다.
```

즉, Vector Search가 추천 후보를 찾는 기술이라면, Explainable Recommendation은 그 후보가 왜 적합한지 설명하는 기술이다. 이 두 기술을 함께 사용하면 추천 결과를 제시하는 데서 끝나지 않고, 사용자가 자신의 선택이 어떻게 반영되었는지 확인하고 다음 선택을 조정할 수 있는 구조로 확장할 수 있다.

---

### Explainable Recommendation의 역할 정리

정리하면 Explainable Recommendation은 다음 역할을 한다.

```
1. 추천 결과가 나온 이유를 설명한다.
2. 사용자 요구와 콘텐츠 정보의 연결 관계를 제시한다.
3. Vector Search 결과를 사용자가 이해할 수 있는 언어로 변환한다.
4. 사용자가 추천 과정을 이해하고 다음 선택을 조정할 수 있게 한다.
```

이러한 기술을 실제로 다루기 위해서는 검색, 임베딩, 클라우드 기반 인덱스 구조에 대한 이해가 필요하다. 이번 콘텐츠의 목적은 새로운 추천 모델을 직접 개발하는 것이 아니라, 기술의 원리와 활용 흐름을 이해하고 이를 데모 구조로 연결하는 것이다. 따라서 다음 단계에서는 Microsoft Learn의 Azure 관련 모듈을 활용하는 이유를 살펴본다.

---

# 4. MS Learn 활용 방향

이번 콘텐츠에서는 기술을 처음부터 모두 직접 구현하는 방식보다, 이미 제공되는 학습 모듈과 클라우드 기반 도구를 활용하는 방식을 중심으로 접근한다. 이유는 명확하다. 이번 기획의 목적은 거대한 추천 모델이나 검색 엔진을 새로 개발하는 데 있지 않다. 중요한 것은 사용자의 막연한 요구를 의미 기반으로 처리하고, 그 결과를 설명 가능한 추천 구조로 연결하는 흐름을 이해하는 것이다.

Microsoft Learn의 Azure 관련 모듈은 이 흐름을 실제 기술 구조로 이해하기 위한 기반 자료로 활용된다. 특히 Azure AI Search, Embedding, 클라우드 기반 검색 인덱스, Explainability 관점은 개별 개념으로만 이해하기보다 하나의 추천 흐름 안에서 함께 다루는 것이 효과적이다.

| 학습 영역 | 프로젝트 내 활용 |
| --- | --- |
| Azure AI Search | 콘텐츠 설명문을 인덱스화하고 Vector Search 구조를 이해 |
| Embedding 기반 검색 | 사용자 요구 프로필과 콘텐츠 설명문을 벡터로 변환하는 흐름 이해 |
| Top-K 검색 결과 반환 | 유사도 기준으로 추천 후보를 추출하는 구조 이해 |
| Responsible AI / Explainability | 추천 결과의 근거 설명과 사용자 통제성 확보 관점 이해 |

MS Learn과 Azure를 활용하는 이유는 다음과 같다.

| 이유 | 설명 |
| --- | --- |
| 공식 학습 자료 | Microsoft가 제공하는 Azure 기반 공식 학습 자료를 통해 기술 흐름을 안정적으로 이해할 수 있음 |
| 빠른 프로토타입 가능 | 이미 제공되는 Azure 기능을 활용해 검색 인덱스와 추천 흐름을 빠르게 구성할 수 있음 |
| 구현 부담 감소 | 검색 엔진, 임베딩, 인덱스 구조를 모두 직접 만들지 않아도 됨 |
| 서비스 조합 학습 | 개별 기술을 조합해 하나의 추천 흐름으로 연결하는 방식을 학습할 수 있음 |
| 콘텐츠 구성 적합성 | 기술 개념과 활용 흐름을 함께 보여주기 좋음 |

직접 추천 모델과 검색 시스템을 만들려면 콘텐츠 데이터셋 수집, 사용자 행동 데이터 확보, 추천 모델 설계, 임베딩 적용, 검색 인덱스 구축, 성능 평가, 배포와 유지보수까지 고려해야 한다. 이번 콘텐츠에서는 이 전체 과정을 직접 완성하는 대신, Azure 기반 기능을 활용해 핵심 원리와 데모 가능성을 보여주는 데 집중한다.

따라서 MS Learn은 단순 참고자료가 아니라, `사용자 요구 프로필 생성 → Embedding → Vector Search → Top-K 결과 반환 → 추천 근거 설명`으로 이어지는 기술 흐름을 이해하기 위한 기준 모델로 활용된다.

이제 앞에서 살펴본 기술이 실제 콘텐츠 추천 흐름에서 어떻게 활용될 수 있는지 예시로 확인한다.

# 5. 기술 활용 데모

## 5-1. 데모의 역할

이 데모는 특정 서비스를 완성형 제품으로 소개하기 위한 것이 아니다. 앞에서 설명한 Vector Search와 Explainable Recommendation이 실제 콘텐츠 추천 흐름에서 어떻게 활용될 수 있는지 직관적으로 보여주기 위한 예시이다.

콘텐츠 추천 상황은 두 기술을 함께 보여주기에 적합하다. 사용자의 요구는 막연하게 나타나고, 콘텐츠 정보는 다양한 표현으로 존재하며, 추천 결과에는 납득 가능한 이유가 필요하기 때문이다.

데모의 핵심은 구현 세부사항보다 **입력, 처리, 출력의 흐름**을 보여주는 데 있다. 사용자의 막연한 요구가 입력 가능한 형태로 구체화되고, 그 요구가 벡터 검색 질의로 변환되며, 콘텐츠 정보와 의미적으로 연결된 뒤, 최종 추천 결과에 근거가 함께 제시되는 흐름을 확인하는 것이다.

```
사용자의 막연한 요구
→ 카드 선택을 통한 취향 입력
→ 사용자 요구 프로필 생성
→ 콘텐츠 설명문 구성
→ Vector Search로 Top-K 후보 추출
→ 추천 근거 생성
```

---

## 5-2. 콘텐츠 추천 데모 시나리오

이 데모에서는 설명을 쉽게 하기 위해 `MealTube`라는 가상의 콘텐츠 추천 상황을 사용한다. 예시는 사용자가 유튜브에서 밥을 먹을 때 볼 영상을 고르지 못하는 상황을 바탕으로 한다.

사용자는 긴 검색어를 직접 입력하지 않고, 자신이 원하는 방향을 간단한 선택으로 표현한다. 이때 카드 선택 방식은 스와이프형 카드 UI를 참고할 수 있다. 사용자는 화면에 나타나는 키워드 카드를 보고, 원하는 방향은 긍정 선택으로 넘기고 원하지 않는 방향은 부정 선택으로 넘기는 방식으로 현재 취향을 직관적으로 입력할 수 있다. 발표에서는 틴더와 유사한 카드 넘김 방식으로 설명하면 청자가 쉽게 이해할 수 있다.

```
긍정 선택:
먹방
짧은 영상
밥친구
가벼운 분위기

부정 선택:
공포
긴 다큐
무거운 분위기
```

데모에서 다루는 데이터는 다음 정도로 단순화할 수 있다.

| 데이터 | 구성 요소 |
| --- | --- |
| 사용자 입력 | 긍정 키워드, 부정 키워드, 선호 분위기, 선호 길이 |
| 콘텐츠 정보 | 제목, 설명, 태그, 요약, 영상 길이 |
| 추천 결과 | 콘텐츠 목록, 유사도 점수, Top-K 순위, 추천 근거 |

이 구조에서 사용자 입력은 하나의 사용자 요구 프로필로 변환된다.

```
밥을 먹으면서 가볍게 볼 수 있는 짧은 먹방 또는 음식 리뷰 콘텐츠를 선호하며,
공포나 긴 다큐처럼 무겁거나 부담스러운 콘텐츠는 피하고 싶음.
```

콘텐츠 정보는 검색 가능한 설명문으로 정리된다.

```
편의점 신상 라면을 먹어보는 짧은 음식 리뷰 영상입니다.
혼밥하거나 밥을 먹을 때 가볍게 보기 좋으며,
먹방, 음식 리뷰, 밥친구, 짧은 영상과 관련된 콘텐츠입니다.
```

이후 사용자 요구 프로필과 콘텐츠 설명문을 각각 벡터로 변환하고, 의미적으로 가까운 콘텐츠를 Top-K 형태로 추천한다. 여기서 Top-K는 유사도 점수가 높은 상위 K개의 콘텐츠를 의미한다.

추천 결과에는 영상 목록만 제시하는 방식보다, 사용자의 선택이 어떻게 반영되었는지 보여주는 설명을 함께 제공하는 것이 중요하다.

```
이 콘텐츠는 사용자가 선택한 ‘먹방’, ‘짧은 영상’, ‘밥친구’와 관련성이 높습니다.
콘텐츠 설명에 ‘혼밥’, ‘음식 리뷰’, ‘짧은 영상’ 정보가 포함되어 있어
밥을 먹으면서 가볍게 볼 수 있는 콘텐츠로 추천되었습니다.
또한 사용자가 피하고 싶다고 선택한 ‘공포’, ‘긴 다큐’와 관련된 요소가 낮아
현재 선택 조건에 적합한 콘텐츠로 판단되었습니다.
```

이 데모를 통해 보여주려는 핵심은 명확하다. Vector Search는 막연한 사용자 요구와 콘텐츠 정보를 의미적으로 연결하고, Explainable Recommendation은 추천 결과가 왜 나왔는지 설명한다. 이를 통해 사용자는 긴 검색어를 직접 만들지 않아도 현재 취향을 입력하고, 추천 결과의 근거까지 확인할 수 있다.

# 6. 정리

이번 콘텐츠는 콘텐츠 플랫폼에서 사용자가 겪는 선택 피로에서 출발했다. 콘텐츠는 많아졌지만, 사용자는 자신이 지금 원하는 콘텐츠를 정확한 검색어로 표현하기 어렵다. 기존의 직접 검색 방식은 명확한 키워드를 요구하고, 일반적인 추천 방식은 사용자의 현재 의도를 직접 반영하거나 추천 이유를 설명하는 데 한계가 있다.

이를 해결하기 위한 기술적 접근으로 Vector Search와 Explainable Recommendation을 살펴보았다. Vector Search는 사용자의 막연한 요구를 문장형 질의로 구성하고, 이를 콘텐츠 설명문과 의미적으로 비교하는 기술이다. Explainable Recommendation은 추천 결과가 사용자의 어떤 선택과 콘텐츠 정보에 기반해 나왔는지 설명하는 기술이다.

전체 흐름은 다음과 같이 정리할 수 있다.

```
카드 선택
→ 사용자 요구 프로필 생성
→ Embedding
→ Vector Search
→ Top-K 추천
→ 추천 근거 생성
```

이 구조를 활용하면 사용자는 긴 검색어를 직접 만들지 않아도 카드 선택을 통해 현재 취향을 입력할 수 있다. 시스템은 이 입력을 사용자 요구 프로필로 변환하고, 콘텐츠 설명문과의 의미적 유사도를 바탕으로 추천 후보를 찾는다. 이후 추천 결과에는 사용자의 선택값과 콘텐츠 정보가 어떻게 연결되었는지에 대한 설명이 함께 제공된다.

결국 이 콘텐츠의 핵심은 콘텐츠를 더 많이 추천하는 데 있지 않다. 사용자의 막연한 요구를 검색 가능한 의미 정보로 바꾸고, 그 결과가 왜 나왔는지 설명할 수 있는 추천 흐름을 설계하는 데 있다.
