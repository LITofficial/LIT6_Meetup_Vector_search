# **1. 우리가 매일 이용하는 것**

우리는 유튜브, 넷플릭스, 쇼츠, 릴스 같은 콘텐츠 플랫폼을 매일 사용합니다. 사용자는 검색창에 정확한 제목을 입력하지 않아도 비슷한 영상, 이어서 볼 만한 콘텐츠, 취향에 맞는 추천 결과를 자연스럽게 마주합니다. 이 경험은 너무 익숙하기 때문에, 그 뒤에서 어떤 기술적 과정이 작동하는지 깊게 생각하지 않는 경우가 많습니다.

그런데 조금만 들여다보면 이 과정은 꽤 흥미롭습니다. 사용자가 직접 “이 영상과 비슷한 영상을 찾아주세요”라고 입력하지 않아도 플랫폼은 관련 있어 보이는 콘텐츠를 계속 제안합니다. 사용자가 정확한 검색어를 입력하지 않아도 이전에 본 콘텐츠, 현재 관심사, 콘텐츠의 주제와 분위기 등을 바탕으로 다음에 볼 만한 결과가 나타납니다.

예를 들어 밥 먹을 때 가볍게 볼 영상이라는 요구와 혼밥할 때 보기 좋은 짧은 먹방 리뷰라는 콘텐츠 설명은 단어가 완전히 같지는 않습니다. 그러나 사람은 두 표현이 비슷한 상황을 가리킨다는 것을 쉽게 이해할 수 있습니다. 둘 다 식사 중에 부담 없이 볼 수 있는 짧고 가벼운 콘텐츠를 떠올리게 하기 때문입니다.

사람은 직관적으로 이해합니다.

“밥 먹을 때 가볍게 볼 영상”

≈ “혼밥할 때 보기 좋은 짧은 먹방 리뷰”

그렇다면, 컴퓨터는 이 비슷함을 어떻게 판단할 수 있을까요?

이번 발표는 이 질문에서 출발합니다. 콘텐츠 검색과 추천 서비스에서는 사용자의 요구와 관련 있어 보이는 후보 콘텐츠를 찾아내는 과정이 있습니다. 실제 추천 시스템은 여러 알고리즘을 섞어서 사용하는데, 저희가 그중 살펴볼 것은 콘텐츠의 의미적 유사성을 계산해 후보를 찾는 방식인 Vector Search입니다. 

Vector Search는 텍스트나 콘텐츠 정보를 숫자 벡터로 변환하고, 벡터 공간에서 의미적으로 가까운 항목을 찾아내는 방식입니다. 같은 단어가 포함되어 있는지만 확인하는 방식보다, 표현이 달라도 의미적으로 가까운 항목을 찾는 데 강점을 가집니다.

이번 발표는 Vector Search의 원리를 청중 참여형 워드클라우드 데모를 통해 보여주는 방식으로 구성됩니다. 청중이 입력한 단어들이 어떻게 검색 가능한 데이터가 되고, 검색어와 의미적으로 가까운 단어들이 어떻게 찾아지는지 화면 변화를 통해 확인합니다. 이후에는 이 원리가 콘텐츠 추천 서비스 아이디어로 어떻게 확장될 수 있는지도 함께 살펴봅니다.

# **2. 느낌에서 데이터로**

사람은 두 표현이 비슷한지 비교적 쉽게 판단할 수 있습니다. 예를 들어 잠들기 전에 틀어놓기 좋은 영상과 잔잔한 분위기의 긴 토크 콘텐츠는 사용하는 단어가 완전히 같지는 않지만, 어느 정도 연결될 수 있습니다. 두 표현 모두 강한 자극보다 편안한 분위기, 배경처럼 틀어놓을 수 있는 콘텐츠, 잠들기 전의 시청 상황과 관련이 있기 때문입니다.

하지만 컴퓨터에게 이 판단은 간단하지 않습니다. 컴퓨터는 문장의 분위기, 맥락, 의도, 상황을 사람처럼 그대로 느끼지 못합니다. 따라서 두 표현이 얼마나 비슷한지 판단하려면, 먼저 텍스트를 비교 가능한 형태로 변환해야 합니다.

이때 필요한 과정이 텍스트를 숫자 벡터로 바꾸는 것입니다. 문장, 단어, 콘텐츠 설명문 같은 텍스트 정보를 벡터로 바꾸면, 컴퓨터는 두 표현을 수치적으로 비교할 수 있습니다. 의미가 비슷한 표현은 벡터 공간에서 가까운 위치에 놓이고, 의미가 다른 표현은 상대적으로 먼 위치에 놓이도록 표현됩니다.

이 관점에서 추천과 검색의 질문은 다음처럼 바뀝니다.

사람의 질문:

이 콘텐츠와 저 콘텐츠가 비슷한가?

컴퓨터가 처리할 수 있는 질문:

두 텍스트 벡터가 벡터 공간에서 얼마나 가까운가?

즉, Vector Search는 사람의 직관적인 “비슷함”을 컴퓨터가 계산할 수 있는 문제로 바꾸는 기술적 접근입니다. 검색어와 데이터가 정확히 일치하는지만 보는 방식에서 한 단계 더 나아가, 텍스트가 담고 있는 의미를 벡터로 표현하고, 그 벡터들이 얼마나 가까운지를 나타내는 유사도를 기준으로 관련 있는 항목을 찾습니다.

물론 실제 대형 플랫폼의 추천 시스템은 사용자 행동 데이터, 콘텐츠 정보, 여러 추천 모델이 결합된 훨씬 복잡한 구조입니다. 이번 발표는 그 전체 구조를 재현하기보다, 그 안에서 활용될 수 있는 핵심 원리 중 하나인 의미 기반 검색 흐름을 작은 데모로 분해해 보여주는 데 집중합니다.

따라서 이후에는 이 흐름을 구성하는 핵심 개념을 차례대로 살펴봅니다. 먼저 텍스트를 벡터로 바꾸는 **Embedding**, 벡터 공간에서 가까운 항목을 찾는 **Vector Search**, 가까움의 정도를 계산하는 **Cosine Similarity**, 유사도 기준 상위 후보를 반환하는 **Top-K Retrieval**을 중심으로 설명합니다.

# **3. 핵심 기술 개념**

앞에서 던진 질문은 결국 하나로 정리됩니다. 컴퓨터가 텍스트의 의미적 비슷함을 계산하려면, 텍스트를 숫자로 바꾸고, 그 숫자 표현들 사이의 거리를 비교해야 합니다.

이 흐름을 이해하기 위해 필요한 개념이 **Embedding, Vector Search, Cosine Similarity, Top-K Retrieval**입니다.

<aside>
⚙

텍스트 데이터 → Embedding → Vector Representation → Similarity Calculation → Vector Search → Top-K Retrieval

</aside>

핵심은 단순합니다. 먼저 텍스트를 숫자 벡터로 바꾸고, 그 벡터들 사이의 유사도를 계산한 뒤, 가장 관련성이 높은 후보를 찾아내는 것입니다.

# **3-1. Embedding**

Embedding은 자연어 텍스트를 숫자 벡터로 변환하는 과정입니다. 컴퓨터는 문장 자체의 의미를 사람처럼 직접 이해하지 못하기 때문에, 텍스트를 계산 가능한 수치 표현으로 바꾸어야 합니다.

예를 들어 다음 문장들은 표현은 다르지만 의미적으로는 비슷합니다.

밥을 먹으면서 가볍게 볼 수 있는 영상

혼밥할 때 보기 좋은 짧은 음식 리뷰

부담 없이 틀어놓기 좋은 먹방 콘텐츠

식사 중 보기 좋은 짧은 예능형 콘텐츠

이 문장들은 같은 단어를 반복하지 않아도 식사 중 가볍게 볼 콘텐츠라는 의미를 공유합니다. Embedding은 이러한 의미적 특징을 벡터 공간에 표현합니다. 의미가 비슷한 문장은 유사한 방향에 놓이고, 의미가 다른 문장은 상대적으로 다른 방향에 놓이도록 표현됩니다.

예를 들어 벡터 공간을 매우 단순화해서 생각하면, 먹방, 음식 리뷰, 혼밥, 가벼운 예능 같은 표현은 서로 유사한 방향으로 배치될 수 있습니다. 반면 범죄 다큐멘터리, 긴 시사 토론, 공포 실화 같은 표현은 사용자의 요구와 다른 방향의 벡터로 나타날 수 있습니다.

따라서 Embedding은 사용자 요구와 콘텐츠 정보를 같은 기준에서 비교할 수 있게 만드는 기반 단계입니다. 텍스트를 벡터로 바꾸어야 그다음 단계에서 유사도를 계산하고, 의미적으로 가까운 항목을 찾을 수 있습니다.

# **3-2. Vector Search**

Vector Search는 텍스트나 문서를 벡터로 변환한 뒤, 벡터 공간에서 유사한 벡터를 찾는 검색 방식입니다. Keyword Search가 주로 “같은 단어가 포함되어 있는가?”를 기준으로 작동한다면, Vector Search는 “의미적으로 가까운가?”를 기준으로 작동합니다.

예를 들어 사용자가 밥 먹을 때 가볍게 볼 수 있는 짧은 영상을 원한다고 했을 때, 콘텐츠 제목이나 설명에 이 문장이 그대로 들어 있지 않아도 관련 콘텐츠를 찾을 수 있습니다. 콘텐츠 설명문에 혼밥, 음식 리뷰, 10분 내외, 가벼운 분위기 같은 정보가 포함되어 있다면, 두 텍스트는 의미적으로 가까운 것으로 판단될 수 있습니다.

Vector Search의 기본 흐름은 다음과 같습니다.

<aside>
⚙

사용자 요구 텍스트 → Embedding → user_vector

콘텐츠 정보 → Embedding → content_vector

user_vector와 content_vector 비교 → 의미적으로 가까운 콘텐츠 검색

</aside>

이 방식의 장점은 사용자가 정확한 검색어를 모르더라도, 문장형 요구나 맥락 정보를 기반으로 관련 콘텐츠를 찾을 수 있다는 점입니다. 사용자가 떠올린 표현과 데이터 안의 표현이 완전히 같지 않아도, 두 표현이 비슷한 의미를 가진다면 검색 결과로 연결될 수 있습니다.

# **3-3. Cosine Similarity**

Vector Search에서 사용자 요구 벡터와 콘텐츠 벡터가 얼마나 유사한지 계산하려면 유사도 측정 방식이 필요합니다. 대표적으로 사용할 수 있는 방식이 **Cosine Similarity**입니다.

Cosine Similarity는 두 벡터가 같은 방향을 향하는 정도를 계산합니다. 값이 높을수록 두 벡터가 의미적으로 가깝다고 볼 수 있습니다.

<aside>
⚙

cosine_similarity(user_vector, content_vector)

→ 사용자 요구와 콘텐츠 설명문의 의미적 가까움

</aside>

여기서 중요한 점은 벡터의 크기보다 방향입니다. 두 벡터가 비슷한 방향을 가리키면, 두 텍스트가 담고 있는 의미도 비슷하다고 해석할 수 있습니다. 따라서 Cosine Similarity는 텍스트의 의미적 유사성을 점수화하는 기준으로 자주 활용됩니다.

예를 들어 사용자 요구 프로필이 밥을 먹으면서 가볍게 볼 수 있는 짧은 먹방 콘텐츠이고, 콘텐츠 설명문이 혼밥할 때 보기 좋은 8분 길이의 음식 리뷰 영상이라면 두 벡터의 방향은 비교적 유사할 수 있습니다. 반대로 콘텐츠 설명문이 45분 길이의 범죄 다큐멘터리라면 사용자의 요구와 의미적으로 멀게 계산될 가능성이 높습니다.

이처럼 Cosine Similarity는 Vector Search 결과를 정렬하고 점수화하는 데 중요한 기준이 됩니다.

# **3-4. Top-K Retrieval**

Vector Search는 모든 콘텐츠를 한 번에 보여주는 방식이 아닙니다. 사용자 요구 벡터와 콘텐츠 벡터 사이의 유사도를 계산한 뒤, 유사도 점수가 높은 상위 K개의 콘텐츠를 반환합니다. 이를 **Top-K Retrieval**이라고 합니다.

<aside>
⚙

사용자 요구 벡터 → 콘텐츠 벡터들과 유사도 비교 → 유사도 점수 계산 → 상위 K개 콘텐츠 반환

</aside>

예를 들어 K를 5로 설정하면, 시스템은 사용자 요구와 의미적으로 가까운 콘텐츠 5개를 후보로 반환합니다. 이 단계는 검색이나 추천 결과의 1차 후보군을 만드는 과정입니다.

Top-K Retrieval이 중요한 이유는 전체 데이터 중에서 관련성이 높은 후보를 빠르게 추려낼 수 있기 때문입니다. 모든 콘텐츠를 사용자에게 보여주는 대신, 사용자 요구와 가까운 후보를 먼저 찾고, 그다음에 필요한 조건을 반영해 최종 결과를 조정할 수 있습니다.

이번 데모에서도 Top-K 개념이 사용됩니다. 발표자가 검색어와 K 값을 입력하면, 시스템은 청중이 입력한 단어들 중 검색어와 의미적으로 가까운 상위 K개의 단어를 반환하고, 해당 단어들을 워드클라우드에서 강조합니다.

# **3-5. Keyword Search와 Vector Search의 차이**

Keyword Search와 Vector Search의 차이는 검색 기준에서 가장 분명하게 드러납니다.

| **구분** | **Keyword Search** | **Vector Search** |
| --- | --- | --- |
| 검색 기준 | 단어의 직접 일치 | 의미적 유사성 |
| 입력 방식 | 짧은 검색어 중심 | 문장, 설명, 맥락 중심 |
| 강점 | 정확한 단어 검색에 강함 | 표현이 달라도 관련 의미 검색 가능 |
| 한계 | 단어가 다르면 놓칠 수 있음 | 임베딩 품질과 데이터 구성에 영향을 받음 |
| 적합한 상황 | 제목, 고유명사, 명확한 키워드 검색 | 요구가 막연하거나 표현이 다양한 검색 |

정확한 제목, 인물명, 사건명처럼 검색어가 분명한 상황에서는 Keyword Search가 효과적입니다. 예를 들어 특정 유튜버 이름이나 영화 제목을 알고 있다면, 같은 단어가 포함된 결과를 찾는 방식이 빠르고 정확할 수 있습니다.

반면 사용자의 요구가 가볍게 볼 만한 것, 잠들기 전에 틀어놓을 것, 밥 먹을 때 보기 좋은 영상처럼 맥락 중심으로 나타나는 상황에서는 Vector Search가 더 적합한 기반이 될 수 있습니다. 이 경우 사용자는 정확한 제목이나 태그를 알고 있는 것이 아니라, 자신이 원하는 분위기와 상황을 표현하고 있기 때문입니다.

실제 검색 시스템에서는 Keyword Search와 Vector Search가 함께 쓰이는 경우가 많습니다. 정확한 고유명사나 제목 검색에는 키워드 검색이 유리하고, 표현이 다양하거나 요구가 문장형으로 주어지는 경우에는 Vector Search가 유리합니다. 따라서 핵심은 두 방식 중 하나가 절대적으로 우월하다는 데 있지 않습니다. 검색하려는 문제의 성격에 따라 적합한 방식을 선택하거나 결합하는 것이 중요합니다.

이번 발표에서는 그중에서도 표현이 달라도 의미적으로 가까운 항목을 찾는 Vector Search의 특징에 집중합니다.

# **3-6. Vector Search의 한계와 설계 포인트**

Vector Search가 모든 문제를 자동으로 해결하는 것은 아닙니다. 검색 품질은 임베딩 모델의 성능뿐만 아니라, 어떤 텍스트를 벡터화하느냐에 크게 영향을 받습니다.

사용자 요구가 너무 짧거나 모호하면 사용자 벡터가 충분한 의미 정보를 담기 어렵습니다. 예를 들어 사용자가 단순히 재밌는 거라고만 입력하면, 이 표현은 너무 넓고 추상적이어서 어떤 콘텐츠와 가까운지 판단하기 어렵습니다.

콘텐츠 정보도 마찬가지입니다. 영상 제목만 벡터화하면 콘텐츠의 분위기, 길이, 시청 상황, 장르 정보가 충분히 반영되지 않을 수 있습니다. 오늘 먹은 음식이라는 제목만으로는 이 영상이 먹방인지, 브이로그인지, 리뷰인지, 예능형 콘텐츠인지 판단하기 어렵습니다.

따라서 Vector Search를 효과적으로 사용하려면 두 가지 설계가 중요합니다.

1. 사용자 요구를 충분한 의미를 담은 문장형 프로필로 구성해야 합니다.

2. 콘텐츠 정보를 제목만이 아니라 설명문, 장르, 분위기, 길이, 시청 상황까지 포함해 구성해야 합니다.

이 두 요소가 제대로 설계되어야 사용자 요구와 콘텐츠 정보가 같은 벡터 공간에서 의미 있게 비교될 수 있습니다. 결국 Vector Search의 성능은 모델만의 문제가 아니라, 어떤 데이터를 어떤 형태로 표현하느냐에 크게 좌우됩니다.

# **4. 청중 참여형 Vector Search 데모**

앞에서 살펴본 것처럼 Vector Search의 핵심은 텍스트를 벡터로 변환하고, 벡터 공간에서 의미적으로 가까운 항목을 찾는 데 있습니다. 이번 데모는 이 원리를 청중이 직접 입력한 단어와 워드클라우드 변화를 통해 확인할 수 있도록 구성했습니다.

이 데모는 완성된 콘텐츠 추천 서비스를 보여주는 화면보다, Embedding, Vector Search, Top-K Retrieval이 실제 입력 데이터와 화면 변화로 어떻게 연결되는지 보여주는 원리 시연에 가깝습니다. 개인화 추천 전체 구조까지 구현하기보다, 의미 기반 검색이 어떻게 작동하는지 직관적으로 보여주는 데 집중합니다.

데모는 Next.js 기반 웹 애플리케이션으로 구성되며, 전체 흐름은 입력 수집 → 저장 → 검색 → 시각화로 나뉩니다. 청중은 QR을 통해 입력 화면에 접속해 단어 또는 짧은 문구를 입력하고, 발표자는 발표자 화면에서 워드클라우드와 검색 결과를 확인합니다.

<aside>
⚙

QR 화면 → 청중 입력 화면 → 입력 저장 API → 발표자 워드클라우드 화면 → 검색 API → Azure Vector Search → Top-K 단어 강조

</aside>

# **4-1. 입력 수집과 워드클라우드 형성**

청중은 입력 화면에서 자신이 떠올리는 주제, 장르, 분위기, 관심사, 콘텐츠 키워드 등을 자유롭게 입력합니다. 입력값은 반드시 정해진 카테고리일 필요가 없습니다. 먹방, 공포, 축구 같은 단어뿐 아니라 편하게 볼 것, 잠들기 전, 밥 먹을 때 보기 좋은 것처럼 짧은 표현도 사용할 수 있습니다.

입력 예시:

먹방 | 공포 | 축구 | 여행 | 편하게 볼 것 | 짧은 리뷰 | 잠들기 전 | 예능 | 힐링

입력된 값은 서버 API를 통해 저장됩니다. 이 과정에서 입력값을 그대로 누적하지 않고, 공백 제거, 길이 제한, 중복 확인과 같은 기본 전처리를 거칩니다. 이를 통해 비어 있는 입력, 지나치게 긴 입력, 반복 입력이 그대로 저장되는 문제를 줄일 수 있습니다.

청중 입력 → 기본 검증 → 정규화 → 전처리 후 저장 → 워드클라우드 반영

검색 전의 워드클라우드는 청중이 입력한 단어와 짧은 표현들을 한 화면에 모아 보여주는 시각화 영역입니다. 이 단계에서는 사용자가 어떤 주제, 장르, 분위기, 관심사를 떠올렸는지 전체적인 입력 흐름을 확인할 수 있습니다. 

일반적인 워드클라우드가 입력된 단어를 시각적으로 모아 보여주는 방식이라면, 이번 데모의 핵심은 이후 검색 과정에서 드러납니다. 단순히 입력값을 나열하는 데서 끝나지 않고, 발표자가 입력한 검색어와 의미적으로 가까운 단어를 찾아 다시 강조하기 때문입니다.

# **4-2. 검색어 입력과 Top-K 검색**

워드클라우드가 형성되면 발표자는 발표자 화면에서 검색어와 Top-K 값을 입력합니다. 검색어는 기존 키워드 검색의 검색어처럼 보일 수 있습니다. 그러나 처리 방식은 다릅니다.

Keyword Search는 같은 문자열이 포함된 항목을 찾는 데 집중합니다. 반면 Vector Search는 검색어와 입력값을 벡터로 변환한 뒤, 벡터 공간에서 가까운 항목을 찾습니다.

<aside>
⚙

검색어 → Embedding → query_vector

청중 입력값 → Embedding → word_vectors

query_vector와 word_vectors 비교 → Top-K 관련 단어 반환

</aside>

예를 들어 발표자가 편하게 볼 것을 검색했을 때, 워드클라우드 안에 정확히 같은 표현이 없어도 힐링, 예능, 잠들기 전, 먹방 같은 단어가 의미적으로 가까운 항목으로 반환될 수 있습니다. 이 장면은 Vector Search가 단순 문자열 일치보다 의미적 유사성을 기준으로 작동한다는 점을 보여주는 핵심 시연입니다.

이때 Top-K 값은 몇 개의 결과를 가져올지 결정합니다. K를 5로 설정하면 검색어와 가장 가까운 5개의 단어가 반환되고, K를 10으로 설정하면 상위 10개의 단어가 반환됩니다. 이를 통해 검색 결과의 범위를 조절할 수 있습니다.

# **4-3. Azure 기반 Vector Search 구조**

Azure 설정이 준비된 경우, 검색은 Azure OpenAI Embedding과 Azure AI Search의 Vector Search를 중심으로 수행됩니다. 여기서 Azure OpenAI Embedding은 검색어와 입력 텍스트를 벡터로 변환하고, Azure AI Search는 벡터화된 데이터를 인덱싱한 뒤 query vector와 가까운 항목을 검색합니다.

<aside>
⚙

Azure OpenAI Embedding

→ 검색어와 입력값을 벡터로 변환

</aside>

<aside>
⚙

Azure AI Search

→ 벡터 인덱싱

→ query vector 기반 유사도 검색

→ Top-K 결과 반환

</aside>

Azure AI Search에서는 입력 단어마다 텍스트와 벡터 값을 함께 저장하는 인덱스를 구성합니다. 이때 텍스트 필드는 실제 단어를 보여주기 위한 정보로 사용되고, vector field는 유사도 검색을 위한 수치 표현으로 사용됩니다.

검색 시에는 발표자가 입력한 검색어를 다시 벡터로 변환하고, 인덱스 안의 vector field와 비교하여 가장 가까운 K개의 항목을 반환합니다. 이 과정은 벡터 공간에서 query vector와 가까운 이웃을 찾는 k-nearest neighbors 검색으로 이해할 수 있습니다.

즉, 데모의 핵심 기술 흐름은 다음과 같이 정리할 수 있습니다.

<aside>
⚙

청중 입력 텍스트 → Embedding → 벡터 인덱스에 저장

발표자 검색어 → Embedding → query vector 생성

query vector와 저장된 벡터 비교 → 의미적으로 가까운 Top-K 단어 반환

</aside>

이 구조를 통해 청중이 입력한 단어는 단순한 텍스트 목록이 아니라, 의미 기반 검색이 가능한 벡터 데이터로 바뀝니다.

# **4-4. 워드클라우드 강조 방식**

검색 결과로 반환된 Top-K 단어는 워드클라우드에서 시각적으로 강조됩니다. 이때 강조는 입력 빈도를 기준으로 단어 크기를 계산하는 방식이 아니라, 검색어와 의미적으로 가까운 단어를 구분해서 보여주는 방식입니다.

<aside>
⚙

검색 전:
청중이 입력한 단어와 짧은 표현을 워드클라우드에 표시함
→ 입력된 키워드들을 한 화면에서 확인할 수 있음
검색 후:
Vector Search를 통해 검색어와 의미적으로 가까운 Top-K 단어를 찾음
→ Top-K에 포함된 단어를 워드클라우드에서 강조 표시함

</aside>

예를 들어 검색어가 편하게 볼 것이고 Top-K 결과가 힐링, 예능, 먹방, 잠들기 전, 여행이라면, 해당 단어들은 워드클라우드 안에서 강조됩니다. 이를 통해 발표자는 검색어와 청중 입력값 사이에서 어떤 단어들이 의미적으로 가깝게 판단되었는지 직관적으로 확인할 수 있습니다.
이때 워드클라우드의 강조는 단순한 디자인 효과가 아니라, 청중이 입력한 데이터가 검색 가능한 후보군이 되고, 검색어와의 의미적 유사도 계산 결과가 다시 화면에 반영되는 과정입니다.
즉, 워드클라우드는 청중 입력값을 모아 보여주는 시각화 영역이면서, Vector Search 결과를 직관적으로 확인할 수 있게 해주는 인터페이스입니다.

# **4-5. 데모의 기술적 의미**

이 데모의 핵심은 사용자가 입력한 표현이 검색 가능한 의미 정보로 바뀔 수 있다는 점입니다. 청중이 입력한 단어는 단순 문자열로 끝나지 않고, 저장·정규화·검색·시각화 과정을 거쳐 검색 대상 데이터가 됩니다.

Azure 기반 검색이 활성화되면 입력값과 검색어는 같은 벡터 공간에서 비교되고, 시스템은 query vector와 가까운 항목을 Top-K 결과로 반환합니다.

<aside>
⚙

워드클라우드 데모:

검색어 → 청중 입력 단어와 의미적 비교 → Top-K 관련 단어 강조

콘텐츠 추천 흐름:

사용자 요구 프로필 → 콘텐츠 설명문과 의미적 비교 → Top-K 추천 후보 반환 → 조건 반영 → 최종 추천 결과 표시

</aside>

따라서 이 데모는 Vector Search의 작동 원리를 축소해서 보여주는 기술 시연입니다. 청중은 워드클라우드의 변화 과정을 통해 Vector Search가 같은 단어의 포함 여부보다 의미적 유사성을 기준으로 관련 항목을 찾는다는 점을 직관적으로 확인할 수 있습니다.

# **4-6. 데모 운영 시 주의사항**

실제 환경변수, API Key, endpoint, 실행 로그, 발표 중 생성된 입력 데이터는 문서나 공유 파일에 포함하지 않습니다. 발표에서는 입력 수집, 저장, 검색, 시각화 흐름만 설명하고, 인증 정보와 실행 데이터는 공유 대상에서 제외합니다.

데모 실행 중 Azure 연결이 불안정하거나 검색 요청에 실패할 가능성도 고려해야 합니다. 이 경우 발표 흐름이 완전히 중단되지 않도록 기본 검색 또는 fallback 로직을 보조 장치로 둘 수 있습니다. 다만 발표의 핵심 설명은 Azure 기반 Vector Search 흐름을 중심으로 진행합니다.

# **5. 아이디어 확장: Catnap**

앞선 워드클라우드 데모가 Vector Search의 기본 원리를 보여주는 작은 시연이라면, **Catnap**은 이 원리가 실제 콘텐츠 추천 경험으로 확장될 수 있음을 보여주는 아이디어 예시입니다.

Catnap은 완성된 상용 서비스를 제안하기 위한 기획보다, 앞선 데모에서 확인한 Vector Search 흐름을 사용자 경험 관점으로 확장한 시안입니다. 사용자가 직접 긴 검색어를 만들지 않아도, 현재 상황과 취향을 카드로 선택하면 시스템은 이를 사용자 요구 프로필로 변환하고 콘텐츠 설명문과 의미적으로 비교할 수 있습니다.

# **5-1. Catnap의 기본 아이디어**

Catnap은 사용자가 콘텐츠를 고르기 어려운 순간에, 현재 의도를 쉽게 표현할 수 있도록 돕는 서비스 아이디어입니다. 사용자는 검색창에 긴 문장을 직접 입력하기보다, 먼저 자신이 콘텐츠를 보려는 상황을 선택합니다.

<aside>
⚙

예시 모드:

밥친구 모드 | 잠들기 전 모드 | 짧게 웃고 싶음 모드 | 가볍게 틀어놓기 모드 | 집중 없이 보기 모드

</aside>

이후 사용자는 카드 선택을 통해 선호 장르, 분위기, 영상 길이, 피하고 싶은 요소를 입력합니다. 예를 들어 틴더와 유사한 카드형 UI를 적용하면, 사용자는 복잡한 설정 화면을 거치지 않고도 쉽고 직관적으로 자신의 현재 취향을 표현할 수 있습니다.

선호 카드: 먹방 | 예능 | 짧은 영상 | 힐링 | 가벼운 분위기

회피 카드: 공포 | 긴 다큐 | 무거운 분위기 | 정치 이슈 | 강한 몰입이 필요한 콘텐츠

카드 선택은 단순한 UI 요소에 머무르지 않습니다. 사용자의 막연한 요구를 구조화된 입력값으로 바꾸는 단계입니다. 이후 시스템은 이 선택값을 바탕으로 사용자 요구 프로필을 생성하고, 이를 벡터 검색에 활용합니다.

# **5-2. 사용자 요구 프로필과 콘텐츠 설명문 비교**

예를 들어 사용자가 다음과 같이 선택했다고 가정해보겠습니다.

<aside>
⚙

상황: 밥친구 모드

선호: 먹방 | 짧은 영상 | 가벼운 분위기 | 예능형 콘텐츠

회피: 공포 | 긴 다큐 | 무거운 분위기

</aside>

시스템은 이 선택값을 그대로 검색에 넣기보다, 의미가 풍부한 사용자 요구 프로필로 변환할 수 있습니다.

사용자는 밥을 먹으면서 부담 없이 볼 수 있는 짧고 가벼운 콘텐츠를 원합니다.

먹방이나 예능형 콘텐츠에 긍정적이며, 공포나 긴 다큐처럼 몰입 부담이 큰 콘텐츠는 피하고 싶어 합니다.

이 사용자 요구 프로필은 Embedding을 통해 벡터화됩니다. 이후 콘텐츠 설명문 벡터와 비교하여 의미적으로 가까운 후보를 찾습니다.

Vector Search의 품질은 콘텐츠 설명문 구성에도 크게 영향을 받습니다. 영상 제목만 벡터화하면 콘텐츠의 분위기나 시청 상황을 충분히 담기 어렵습니다. 따라서 Catnap에서는 콘텐츠 정보를 다음과 같이 구성할 수 있습니다.

<aside>
⚙

콘텐츠 제목 | 장르 | 영상 길이 | 분위기 | 추천 시청 상황 | 주요 키워드 | 간단한 설명문

</aside>

예를 들어 하나의 콘텐츠는 다음처럼 표현될 수 있습니다.

<aside>
⚙

제목: 혼밥할 때 보기 좋은 편의점 신상 먹방 리뷰

장르: 먹방, 리뷰

길이: 8분

분위기: 가벼움, 예능형, 부담 없음

추천 상황: 식사 중, 짧은 휴식 시간

</aside>

설명문: 혼밥할 때 가볍게 틀어놓기 좋은 짧은 음식 리뷰 영상입니다. 편의점 신상 메뉴를 예능적인 분위기로 소개하며, 긴 집중 없이 부담 없이 볼 수 있습니다.

이처럼 콘텐츠 설명문이 충분히 구성되면, 사용자 요구 프로필과 콘텐츠 설명문을 같은 벡터 공간에서 비교할 수 있습니다. 사용자가 정확한 제목을 몰라도, 현재 상황과 의미적으로 가까운 콘텐츠를 찾을 가능성이 높아집니다.

# **5-3. 조건 기반 재정렬과 확장 가능성**

Vector Search는 의미적으로 가까운 후보를 찾는 데 강점을 가집니다. 다만 추천 서비스로 확장할 경우, 유사도 점수만으로 최종 결과를 결정하면 부족할 수 있습니다. 사용자는 원하는 조건뿐 아니라 피하고 싶은 조건도 함께 가지고 있기 때문입니다.

예를 들어 사용자가 먹방, 짧은 영상, 가벼운 분위기를 선호하고, 공포, 긴 다큐, 무거운 분위기를 피하고 싶다고 선택했다면, 유사도 점수 외에도 조건 반영이 필요합니다.

이를 위해 최종 점수를 다음처럼 조정할 수 있습니다.

<aside>
⚙

final_score =

vector_similarity + positive_match_score - negative_match_penalty - duration_penalty

</aside>

여기서 vector_similarity는 사용자 요구 벡터와 콘텐츠 벡터 사이의 의미적 유사도입니다. positive_match_score는 사용자가 선호한 조건과 콘텐츠 정보가 연결될 때 더해지는 점수이고, negative_match_penalty는 사용자가 피하고 싶은 조건과 관련성이 높을 때 차감되는 점수입니다. duration_penalty는 사용자가 짧은 영상을 선호했는데 콘텐츠가 지나치게 긴 경우 적용할 수 있습니다.

조건 기반 재정렬은 Vector Search 자체의 필수 구성 요소라기보다, Vector Search로 찾은 후보를 서비스 목적에 맞게 다시 정렬하는 후처리 단계입니다. 즉, Vector Search가 의미적으로 가까운 후보군을 먼저 찾고, 조건 기반 재정렬은 사용자의 현재 기호와 회피 조건을 반영해 최종 추천 결과를 조정합니다.

<aside>
⚙

1차 검색:

사용자 요구 프로필 벡터 → 콘텐츠 설명문 벡터들과 유사도 비교 → Top-K 후보 반환

최종 조정:

선호 조건 반영 | 회피 조건 차감 | 영상 길이 조건 반영

→ 최종 추천 결과 정렬

</aside>

이 구조를 통해 Catnap은 단순히 비슷한 콘텐츠를 찾는 수준에서 더 나아가, 사용자의 현재 상황과 조건을 반영한 추천 경험으로 확장될 수 있습니다.

앞선 워드클라우드 데모가 청중이 입력한 단어와 검색어를 벡터 공간에서 비교하는 원리 시연이었다면, Catnap은 같은 원리를 콘텐츠 추천 경험으로 확장한 예시입니다.

<aside>
⚙

워드클라우드 데모:

청중 입력 단어 → 검색어와 의미적 비교 → Top-K 단어 강조

Catnap:

사용자 상황·취향 선택 → 사용자 요구 프로필 생성 → 콘텐츠 설명문과 의미적 비교 → Top-K 추천 후보 반환 → 조건 기반 재정렬

</aside>

따라서 Catnap은 발표의 핵심 기술을 서비스 아이디어로 연결하는 역할을 합니다. Vector Search가 실제 사용자 경험 안에서 어떤 방식으로 활용될 수 있는지 보여주는 확장 예시라고 볼 수 있습니다.

# **6. 정리**

이번 발표는 우리가 매일 경험하는 추천 기술에서 출발했습니다. 유튜브, 넷플릭스, 쇼츠, 릴스 같은 플랫폼은 사용자가 정확한 검색어를 입력하지 않아도 관련 콘텐츠를 계속 제안합니다. 이때 중요한 질문은 “컴퓨터가 비슷하다는 것을 어떻게 판단할 수 있을까?”입니다.

이 질문을 설명하기 위해 Vector Search의 핵심 흐름을 살펴보았습니다. 텍스트는 Embedding을 통해 숫자 벡터로 변환되고, Vector Search는 벡터 공간에서 의미적으로 가까운 항목을 찾습니다. Cosine Similarity는 두 벡터의 유사함을 점수화하고, Top-K Retrieval은 유사도 기준 상위 후보를 반환합니다.

청중 참여형 워드클라우드 데모는 이 원리를 실제 화면에서 보여주는 시연입니다. 청중이 입력한 단어는 검색 가능한 데이터가 되고, 발표자가 검색어를 입력하면 의미적으로 가까운 Top-K 단어가 워드클라우드에서 강조됩니다. 이를 통해 Vector Search가 같은 단어의 포함 여부보다 의미적 유사성을 기준으로 작동한다는 점을 직관적으로 확인할 수 있습니다.

Catnap은 이 기술 흐름을 콘텐츠 추천 경험으로 확장한 아이디어입니다. 사용자는 현재 상황과 취향을 카드로 선택하고, 시스템은 이를 사용자 요구 프로필로 변환합니다. 이후 콘텐츠 설명문과 벡터 공간에서 비교하여 의미적으로 가까운 후보를 찾고, 선호 조건과 회피 조건을 반영해 최종 결과를 조정할 수 있습니다.

결국 이번 발표의 핵심은 다음과 같이 정리할 수 있습니다.

컴퓨터는 문장의 느낌을 그대로 이해하지 못합니다.

그래서 텍스트를 숫자 벡터로 바꾸고,

벡터 공간에서 가까운 항목을 찾습니다.

이 과정을 통해 검색어와 데이터가 정확히 일치하지 않아도

의미적으로 관련 있는 결과를 찾을 수 있습니다.

Vector Search는 이러한 의미 기반 검색을 가능하게 하는 핵심 기술 중 하나입니다.

이번 발표에서는 이 원리를 복잡한 추천 시스템 전체로 다루기보다, 청중이 직접 입력하고 화면 변화를 확인할 수 있는 작은 데모로 풀어봅니다. 이를 통해 Vector Search가 어떤 방식으로 의미 기반 검색을 가능하게 하는지, 그리고 콘텐츠 추천 경험으로 어떻게 확장될 수 있는지 이해하는 것이 목표입니다.
