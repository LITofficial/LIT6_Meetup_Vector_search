# 0. 흐름 요약

```markdown
1. 문제 상황 인식 및 공감대 형성
   - 사용자는 밥을 먹을 때, 씻을 때, 잠들기 전처럼 가볍게 시간을 보내고 싶은 순간에 “무언가 흥미로운” 동영상을 찾는다.
   - 하지만 추천 화면은 마음에 들지 않고, 직접 검색하기에는 떠오르는 주제도 없다.
   - 이 상황의 핵심은 지금 원하는 콘텐츠가 아직 검색어로 정리될 만큼 명확하지 않다는 점이다.

2. 기존 검색·추천 방식의 한계
   - 키워드 검색은 사용자가 현재 요구에 맞는 단어를 직접 떠올리고 조합해야 한다.
   - 기존 추천은 현재 분위기, 길이, 장르, 피하고 싶은 요소 등 사용자의 요구를 즉시 반영하기 어렵다.
   - 따라서 필요한 것은 막연한 요구를 간단한 선택으로 표현하고, 이를 추천 과정에 반영할 수 있는 구조이다.

3. 핵심 기술 개념
   - Embedding은 텍스트를 의미 기반 벡터로 변환한다.
   - Vector Search는 벡터 공간에서 의미적으로 가까운 항목을 찾는다.
   - Cosine Similarity는 사용자 요구 벡터와 콘텐츠 벡터의 가까움을 점수화한다.
   - Top-K Retrieval은 유사도 기준 상위 K개의 후보를 반환한다.
   - 조건 기반 재정렬은 선호 조건과 회피 조건을 반영해 최종 결과를 조정한다.

4. 청중 참여형 Vector Search 데모
   - 청중이 QR로 접속해 자유롭게 단어를 입력한다.
   - 입력값은 서버 API를 통해 저장되고, 동일 표현이 반복되면 count가 증가한다.
   - 기본 워드클라우드는 입력 빈도를 바탕으로 형성된다.
   - 발표자가 검색어와 Top-K 값을 입력하면 검색 API가 실행된다.
   - Azure 설정이 준비된 경우, 임베딩 모델이 검색어와 입력 단어를 벡터로 변환하고 Azure AI Search가 벡터 인덱스를 기준으로 유사 항목을 검색한다.
   - Azure 연결이 없거나 실패한 경우, fallback 검색을 통해 발표 흐름을 유지한다.
   - 검색 결과로 반환된 Top-K 단어는 워드클라우드에서 더 크게 강조된다.

5. 아이디어 확장 : Catnap
   - 워드클라우드 데모에서 확인한 의미 기반 검색 흐름을 실제 콘텐츠 추천 경험으로 확장한 아이디어 시안이다.
   - 사용자는 “지금 모드”와 카드 선택을 통해 현재 취향과 회피 조건을 입력한다.
   - 시스템은 선택값을 사용자 요구 프로필로 변환하고, 콘텐츠 설명문과 벡터 공간에서 비교한다.
   - 이후 Top-K 추천 후보를 반환하고, 조건 기반 재정렬을 통해 최종 결과를 조정한다.

6. 정리
   - 이 콘텐츠의 핵심은 사용자의 막연한 요구를 검색 가능한 의미 정보로 바꾸는 데 있다.
   - Vector Search는 사용자 요구와 콘텐츠 설명문을 의미적으로 비교해 추천 후보를 찾는다.
   - 청중 참여형 데모는 이 원리를 실제 화면에서 보여주고, Catnap은 이 기술 흐름을 사용자 경험으로 확장한 서비스 아이디어 시안이다.
```

# 1. 기획 배경: 콘텐츠는 많지만 선택은 어려운 상황

콘텐츠 플랫폼은 이미 일상 속에 깊게 들어와 있다. 유튜브, 넷플릭스, 쇼츠, 릴스처럼 우리는 언제든지 원하는 영상을 볼 수 있는 환경에 있다. 그러나 선택지가 많아졌다고 해서 콘텐츠 선택이 쉬워진 것은 아니다. 오히려 콘텐츠가 많아질수록 사용자는 “뭘 봐야 하지?”라는 선택 피로를 더 자주 경험한다.

이 문제는 특히 가볍게 시간을 보내고 싶은 순간에 더 분명하게 드러난다. 밥을 먹을 때, 씻을 때, 잠들기 전 침대에 누웠을 때, 또는 별다른 목적 없이 쉬고 싶을 때 사용자는 시간을 때울 “무언가 흥미로운” 동영상을 찾는다. 하지만 오늘따라 추천 화면은 마음에 들지 않고, 직접 검색을 하자니 귀찮은 데다 딱히 떠오르는 주제도 없다.

```markdown
유튜브를 켰는데 막상 볼 영상이 없다.
밥을 먹으면서 가볍게 볼 영상을 찾고 싶은데 뭘 검색해야 할지 모르겠다.
짧고 재밌는 콘텐츠를 보고 싶은데 정확한 검색어가 떠오르지 않는다.
추천 화면을 계속 넘기지만 지금 보고 싶은 영상이 잘 나오지 않는다.
```

이 상황의 핵심은 사용자가 자신이 지금 원하는 콘텐츠를 검색 가능한 형태로 표현하기 어렵다는 점이다. 사용자는 “가볍게 볼 만한 것”, “시간 때울 수 있는 것”, “무언가 흥미로운 것”, “부담 없이 틀어놓을 수 있는 영상” 같은 막연한 요구를 가지고 있다. 그러나 이런 요구는 검색창에 그대로 입력하기 애매하다. 검색창은 구체적인 키워드를 요구하지만, 사용자의 실제 요구는 아직 명확한 키워드로 정리되지 않은 상태이기 때문이다.

따라서 이 발표는 다음 질문에서 출발한다.

```markdown
사용자의 막연한 요구를 어떻게 검색 가능한 형태로 바꿀 수 있을까?
정확한 검색어 없이도 의미적으로 가까운 콘텐츠를 찾을 수 있을까?
사용자의 현재 분위기와 회피 조건을 추천 과정에 직접 반영할 수 있을까?
```

이 발표의 목표는 사용자의 막연한 현재 요구를 간단한 선택이나 표현으로 입력받고, 이를 의미 기반 검색이 가능한 사용자 요구 프로필로 변환한 뒤, 콘텐츠 설명문과 비교하여 적합한 추천 후보를 찾는 기술 흐름을 소개하는 것이다.

# 2. 기존 검색·추천 방식의 한계

사용자가 콘텐츠를 찾는 방식은 크게 직접 검색과 플랫폼 추천으로 나눌 수 있다. 직접 검색은 명확한 검색어가 있을 때 효과적이다. 사용자가 특정 채널명, 콘텐츠 제목, 인물명, 사건명처럼 분명한 단서를 알고 있다면 검색창은 매우 강력한 도구가 된다. 그러나 “밥 먹을 때 가볍게 볼 영상”, “시간 때울 만한 흥미로운 영상”처럼 요구가 막연한 상황에서는 검색어를 만드는 것 자체가 부담이 된다.

키워드 기반 탐색은 사용자가 먼저 자신의 요구를 언어로 정리해야 한다. 사용자의 요구가 `먹방`, `예능`, `짧은 영상`처럼 명확한 키워드로 정리될 때는 문제가 적다. 하지만 실제 사용자의 요구는 상황, 분위기, 길이, 장르, 회피 조건이 섞인 형태로 나타나는 경우가 많다. 이때 키워드 검색은 현재 요구에 적합한 단어를 떠올리고 조합해야 하므로, 콘텐츠를 고르는 과정 자체를 번거롭게 만들 수 있다.

플랫폼 추천은 사용자가 직접 검색하지 않아도 콘텐츠를 제시한다는 점에서 편리하다. 그러나 일반적인 추천 화면은 사용자의 현재 의도를 명시적으로 입력받는 구조가 약하다. 평소에는 경제 영상을 자주 보던 사용자도 지금은 밥을 먹으면서 볼 가벼운 영상을 원할 수 있다. 사용자의 취향은 고정되어 있지 않고, 상황에 따라 계속 달라지지만, 기존 추천 화면 안에서는 지금 원하는 분위기, 영상 길이, 장르, 피하고 싶은 요소를 즉시 반영하기 어렵다.

| 기존 방식 | 한계 |
| --- | --- |
| 직접 검색 | 사용자가 먼저 요구를 검색어로 정리해야 함 |
| 키워드 기반 탐색 | 표현이 다르면 의미적으로 가까운 콘텐츠를 놓칠 수 있음 |
| 플랫폼 추천 | 사용자의 현재 분위기와 회피 조건을 즉시 반영하기 어려움 |
| 추천 화면 탐색 | 사용자가 원하는 콘텐츠가 나올 때까지 수동적으로 넘겨야 함 |

따라서 필요한 것은 추천 결과를 더 많이 보여주는 기능을 넘어선다. 사용자의 막연한 요구를 간단한 선택이나 표현으로 입력받고, 이를 검색 가능한 의미 정보로 변환한 뒤, 추천 과정에 직접 반영할 수 있는 구조가 필요하다.

# 3. 핵심 기술 개념

이번 콘텐츠의 핵심 기술은 사용자의 막연한 요구를 의미 기반 검색이 가능한 형태로 바꾸고, 콘텐츠 정보와 비교하여 적합한 후보를 찾는 데 있다. 이를 위해 Embedding, Vector Search, Cosine Similarity, Top-K Retrieval, 조건 기반 재정렬의 흐름을 이해할 필요가 있다.

```markdown
텍스트 데이터
→ Embedding
→ Vector Representation
→ Similarity Calculation
→ Vector Search
→ Top-K Retrieval
→ Condition-based Reranking
```

---

## 3-1. Embedding

Embedding은 자연어 텍스트를 숫자 벡터로 변환하는 과정이다. 컴퓨터는 문장 자체의 의미를 그대로 비교할 수 없기 때문에, 텍스트를 수치화된 벡터 표현으로 바꾸어야 한다.

예를 들어 다음 문장들은 표현은 다르지만 의미적으로는 비슷하다.

```markdown
밥을 먹으면서 가볍게 볼 수 있는 영상
혼밥할 때 보기 좋은 짧은 음식 리뷰
부담 없이 틀어놓기 좋은 먹방 콘텐츠
식사 중 보기 좋은 짧은 예능형 콘텐츠
```

이 문장들은 같은 단어를 반복하지 않아도 “식사 중 가볍게 볼 콘텐츠”라는 의미를 공유한다. Embedding은 이러한 의미적 특징을 벡터 공간에 표현한다. 의미가 비슷한 문장은 가까운 위치에 놓이고, 의미가 다른 문장은 상대적으로 먼 위치에 놓이도록 표현된다.

따라서 Embedding은 사용자 요구와 콘텐츠 정보를 같은 기준에서 비교할 수 있게 만드는 기반 단계이다.

---

## 3-2. Vector Search

Vector Search는 텍스트나 문서를 벡터로 변환한 뒤, 벡터 공간에서 가까운 항목을 찾는 검색 방식이다. 기존 Keyword Search가 “같은 단어가 포함되어 있는가?”를 기준으로 작동한다면, Vector Search는 “의미적으로 가까운가?”를 기준으로 작동한다.

예를 들어 사용자가 `밥 먹을 때 가볍게 볼 수 있는 짧은 영상`을 원한다고 했을 때, 콘텐츠 제목이나 태그에 이 문장이 그대로 들어 있지 않아도 관련 콘텐츠를 찾을 수 있다. 콘텐츠 설명문에 `혼밥`, `음식 리뷰`, `10분 내외`, `가벼운 분위기` 같은 정보가 포함되어 있다면, 두 텍스트는 의미적으로 가까운 것으로 판단될 수 있다.

Vector Search의 기본 흐름은 다음과 같다.

```markdown
사용자 요구 텍스트
→ Embedding
→ user_vector

콘텐츠 설명문
→ Embedding
→ content_vector

user_vector와 content_vector 비교
→ 의미적으로 가까운 콘텐츠 검색
```

이 방식의 장점은 사용자가 정확한 검색어를 모르더라도, 문장형 요구나 맥락 정보를 기반으로 관련 콘텐츠를 찾을 수 있다는 점이다.

---

## 3-3. Cosine Similarity

Vector Search에서 사용자 요구 벡터와 콘텐츠 벡터가 얼마나 가까운지 계산하려면 유사도 측정 방식이 필요하다. 데모에서는 대표적으로 Cosine Similarity를 사용할 수 있다.

Cosine Similarity는 두 벡터가 같은 방향을 향하는 정도를 계산한다. 값이 높을수록 두 벡터가 의미적으로 가깝다고 볼 수 있다.

```markdown
cosine_similarity(user_vector, content_vector)
→ 사용자 요구와 콘텐츠 설명문의 의미적 가까움
```

예를 들어 사용자 요구 프로필이 “밥을 먹으면서 가볍게 볼 수 있는 짧은 먹방 콘텐츠”이고, 콘텐츠 설명문이 “혼밥할 때 보기 좋은 8분 길이의 음식 리뷰 영상”이라면 두 벡터의 방향은 비교적 가까울 수 있다. 반대로 콘텐츠 설명문이 “45분 길이의 범죄 다큐멘터리”라면 사용자의 요구와 의미적으로 멀게 계산될 가능성이 높다.

이처럼 Cosine Similarity는 Vector Search 결과를 점수화하는 기준으로 활용된다.

---

## 3-4. Keyword Search와 Vector Search의 차이

Keyword Search와 Vector Search의 차이는 검색 기준에서 가장 분명하게 드러난다.

| 구분 | Keyword Search | Vector Search |
| --- | --- | --- |
| 검색 기준 | 단어의 직접 일치 | 의미적 유사성 |
| 입력 방식 | 짧은 검색어 중심 | 문장, 설명, 맥락 중심 |
| 강점 | 정확한 단어 검색에 강함 | 표현이 달라도 관련 의미 검색 가능 |
| 한계 | 단어가 다르면 놓칠 수 있음 | 임베딩 품질과 데이터 구성에 영향을 받음 |
| 적합한 상황 | 정확한 키워드가 있을 때 | 요구가 막연하거나 표현이 다양할 때 |

정확한 제목, 인물명, 사건명처럼 검색어가 분명한 상황에서는 Keyword Search가 효과적이다. 그러나 사용자의 요구가 “가볍게 볼 만한 것”, “시간 때울 만한 것”, “밥 먹을 때 틀어놓을 것”처럼 막연하게 나타나는 상황에서는 Vector Search가 더 적합한 기반이 된다.

---

## 3-5. Top-K Retrieval

Vector Search는 모든 콘텐츠를 한 번에 보여주는 방식이 아니다. 사용자 요구 벡터와 콘텐츠 벡터 사이의 유사도를 계산한 뒤, 유사도 점수가 높은 상위 K개의 콘텐츠를 반환한다. 이를 Top-K Retrieval이라고 한다.

```markdown
사용자 요구 벡터
→ 콘텐츠 벡터들과 유사도 비교
→ 유사도 점수 계산
→ 상위 K개 콘텐츠 반환
```

예를 들어 K를 5로 설정하면, 시스템은 사용자 요구와 의미적으로 가까운 콘텐츠 5개를 후보로 반환한다. 이 단계는 추천 결과의 1차 후보군을 만드는 과정이다.

Top-K Retrieval은 추천 흐름에서 중요한 역할을 한다. 전체 콘텐츠를 모두 보여주는 대신, 사용자 요구와 가까운 후보만 추려내기 때문이다. 이후에는 선호 조건, 회피 조건, 길이 조건 등을 반영해 최종 결과를 조정할 수 있다.

---

## 3-6. 조건 기반 재정렬

Vector Search로 얻은 유사도 점수만으로 추천 결과를 결정하면 부족할 수 있다. 사용자는 원하는 조건뿐 아니라 피하고 싶은 조건도 함께 가지고 있기 때문이다.

예를 들어 사용자가 `먹방`, `짧은 영상`, `가벼운 분위기`를 선호하고, `공포`, `긴 다큐`, `무거운 분위기`를 피하고 싶다고 선택했다면, 유사도 점수 외에도 조건 반영이 필요하다.

이를 위해 최종 점수를 다음처럼 조정할 수 있다.

```markdown
final_score = vector_similarity
            + positive_match_score
            - negative_match_penalty
            - duration_penalty
```

여기서 `vector_similarity`는 사용자 요구 벡터와 콘텐츠 벡터 사이의 의미적 유사도이다. `positive_match_score`는 사용자가 선호한 조건과 콘텐츠 정보가 연결될 때 더해지는 점수이고, `negative_match_penalty`는 사용자가 피하고 싶은 조건과 관련성이 높을 때 차감되는 점수이다. `duration_penalty`는 사용자가 짧은 영상을 선호했는데 콘텐츠가 지나치게 긴 경우 적용할 수 있다.

즉, Vector Search는 의미적으로 가까운 후보를 찾고, 조건 기반 재정렬은 사용자의 현재 기호와 회피 조건을 반영해 최종 추천 결과를 조정한다.

---

## 3-7. Vector Search의 한계와 설계 포인트

Vector Search가 모든 문제를 자동으로 해결하는 것은 아니다. 검색 품질은 임베딩 모델의 성능뿐 아니라, 어떤 텍스트를 벡터화하느냐에 크게 영향을 받는다.

사용자 요구가 너무 짧거나 모호하면 사용자 벡터가 충분한 의미 정보를 담기 어렵다. 콘텐츠 설명문도 마찬가지다. 영상 제목만 벡터화하면 콘텐츠의 분위기, 길이, 시청 상황, 장르 정보가 충분히 반영되지 않을 수 있다.

따라서 Vector Search를 효과적으로 사용하려면 두 가지 설계가 중요하다.

```markdown
1. 사용자 요구를 문장형 프로필로 구성한다.
2. 콘텐츠 정보를 충분한 설명문으로 구성한다.
```

이 두 요소가 제대로 설계되어야 사용자 요구와 콘텐츠 정보가 같은 벡터 공간에서 의미 있게 비교될 수 있다.

---

# 4. 청중 참여형 Vector Search 데모

앞에서 살펴본 기술 개념을 실제 화면에서 보여주기 위해, 청중 참여형 워드클라우드 데모를 구성한다. 이 데모는 완성된 콘텐츠 추천 서비스를 보여주는 화면이 아니라, Embedding, Vector Search, Top-K Retrieval이 실제 입력 데이터와 화면 변화로 어떻게 연결되는지 보여주는 원리 시연이다. 조건 기반 재정렬은 이후 Catnap 확장 흐름에서 적용되는 요소로 두고, 여기서는 의미 기반 검색이 화면에서 어떻게 드러나는지에 집중한다.

데모는 Next.js 기반 웹 애플리케이션으로 구성되며, 전체 흐름은 `입력 수집 → 저장 → 검색 → 시각화`로 나뉜다. 청중은 QR을 통해 입력 화면에 접속해 단어 또는 짧은 문구를 입력하고, 발표자는 발표자 화면에서 워드클라우드와 검색 결과를 확인한다.

```markdown
QR 화면
→ 청중 입력 화면
→ 입력 저장 API
→ 발표자 워드클라우드 화면
→ 검색 API
→ Azure Vector Search 또는 fallback 검색
→ Top-K 단어 강조
```

---

## 4-1. 입력 수집과 워드클라우드 형성

청중은 입력 화면에서 자신이 떠올리는 주제, 장르, 분위기, 관심사, 콘텐츠 키워드 등을 자유롭게 입력한다. 입력값은 반드시 정해진 카테고리일 필요가 없으며, `먹방`, `공포`, `축구` 같은 단어뿐 아니라 `편하게 볼 것`, `잠들기 전`, `밥 먹을 때 보기 좋은 것`처럼 짧은 표현도 사용할 수 있다.

```markdown
입력 예시:
먹방
공포
축구
여행
편하게 볼 것
짧은 리뷰
잠들기 전
예능
힐링
```

입력된 값은 서버 API를 통해 저장된다. 이때 입력값을 그대로 쌓기보다, 공백 제거, 길이 제한, 중복 확인 같은 기본 처리를 거친다. 같은 표현이 반복해서 들어오면 새로운 항목을 계속 추가하는 대신 기존 항목의 `count`를 증가시킨다.

```markdown
청중 입력
→ 기본 검증
→ 정규화
→ 신규 저장 또는 count 증가
→ 워드클라우드 반영
```

따라서 검색 전의 워드클라우드는 청중의 입력 분포를 보여주는 빈도 기반 시각화로 볼 수 있다. 여러 명이 같은 단어를 입력하면 해당 단어가 더 크게 보이고, 적게 입력된 단어는 상대적으로 작게 표시된다.

---

## 4-2. 검색어 입력과 Top-K 검색

워드클라우드가 형성되면 발표자는 발표자 화면에서 검색어와 Top-K 값을 입력한다. 검색어는 기존 키워드 검색의 검색어처럼 보일 수 있다. 그러나 처리 방식은 다르다. Keyword Search가 같은 문자열이 포함된 항목을 찾는 데 집중한다면, Vector Search는 검색어와 입력값을 벡터로 변환한 뒤 벡터 공간에서 가까운 항목을 찾는다.

```markdown
검색어
→ Embedding
→ query_vector

청중 입력값
→ Embedding
→ word_vectors

query_vector와 word_vectors 비교
→ Top-K 관련 단어 반환
```

예를 들어 발표자가 `편하게 볼 것`을 검색했을 때, 워드클라우드 안에 정확히 같은 표현이 없어도 `힐링`, `예능`, `잠들기 전`, `먹방` 같은 단어가 의미적으로 가까운 항목으로 반환될 수 있다. 이 장면은 Vector Search가 단순 문자열 일치보다 의미적 유사성을 기준으로 작동한다는 점을 보여주는 핵심 시연이다.

---

## 4-3. Azure 기반 Vector Search

Azure 설정이 준비된 경우, 검색은 Azure OpenAI Embedding과 Azure AI Search의 Vector Search를 중심으로 수행된다. 여기서 임베딩 모델은 검색어와 입력 텍스트를 벡터로 변환하고, Azure AI Search는 생성된 벡터를 인덱싱한 뒤 query vector와 가까운 항목을 검색한다.

```markdown
Azure OpenAI Embedding
→ 검색어와 입력값을 벡터로 변환

Azure AI Search
→ 벡터 인덱싱
→ query vector 기반 유사도 검색
→ Top-K 결과 반환
```

---

## 4-4. 워드클라우드 강조 방식

검색 결과로 반환된 Top-K 단어는 워드클라우드에서 더 크게 강조된다. 검색 전의 단어 크기가 주로 입력 빈도에 기반한다면, 검색 후에는 입력 빈도에 검색 결과 boost가 더해져 최종 크기가 조정된다.

```markdown
검색 전:
count 기반 weight
→ 많이 입력된 단어가 크게 표시됨

검색 후:
count 기반 weight
+ Top-K 검색 결과 boost
→ 검색어와 관련 있는 단어가 더 크게 강조됨
```

예를 들어 검색어가 `편하게 볼 것`이고 Top-K 결과가 `힐링`, `예능`, `먹방`, `잠들기 전`, `여행`이라면, 해당 단어들은 워드클라우드에서 더 크게 보인다. 반대로 검색어와 관련성이 낮은 단어는 상대적으로 작게 남는다.

이때 워드클라우드의 크기 변화는 단순한 디자인 효과가 아니다. 청중이 입력한 데이터가 검색 가능한 후보군이 되고, 검색어와의 관련도 계산 결과가 다시 시각적 강조로 반영되는 과정이다. 즉, 워드클라우드는 입력 데이터의 분포와 Vector Search 결과를 함께 보여주는 인터페이스라고 볼 수 있다.

---

## 4-5. 데모의 기술적 의미

이 데모의 핵심은 사용자가 입력한 표현이 검색 가능한 의미 정보로 바뀔 수 있다는 점이다. 청중이 입력한 단어는 단순 문자열로 끝나지 않고, 저장·정규화·검색·시각화 과정을 거쳐 검색 대상 데이터가 된다. Azure 기반 검색이 활성화되면 입력값과 검색어는 같은 벡터 공간에서 비교되고, 시스템은 query vector와 가까운 항목을 Top-K 결과로 반환한다.

```markdown
워드클라우드 데모:
검색어
→ 청중 입력 단어와 의미적 비교
→ Top-K 관련 단어 강조

콘텐츠 추천 흐름:
사용자 요구 프로필
→ 콘텐츠 설명문과 의미적 비교
→ Top-K 추천 후보 반환
→ 조건 기반 재정렬
→ 최종 추천 결과 표시
```

따라서 이 데모는 Catnap에 적용될 수 있는 핵심 검색 원리를 축소해서 보여주는 기술 시연이다. 청중은 워드클라우드의 변화 과정을 통해 Vector Search가 같은 단어의 포함 여부보다 의미적 유사성을 기준으로 관련 항목을 찾는다는 점을 직관적으로 확인할 수 있다.

보안 측면에서는 실제 환경변수, API Key, endpoint, 실행 로그, 발표 중 생성된 입력 데이터가 문서나 공유 파일에 포함되지 않도록 관리해야 한다. 기술 구조를 설명할 때는 입력 수집, 저장, 검색, 시각화 흐름만 다루고, 인증 정보나 실행 중 생성된 데이터는 공유 대상에서 제외하는 것이 원칙이다.

# 5. 아이디어 확장 : Catnap

앞선 워드클라우드 데모는 청중이 입력한 단어와 발표자의 검색어를 벡터 공간에서 비교하고, 의미적으로 가까운 Top-K 항목을 시각적으로 강조하는 원리 시연이다. 이 흐름을 콘텐츠 추천 경험으로 확장하면, 사용자가 정확한 검색어를 만들지 못하는 순간에도 현재 상황과 취향을 입력값으로 바꾸고, 이를 벡터 검색 기반 추천으로 연결할 수 있다. 이러한 확장 방향의 예시로 우리는 ‘Catnap’이라는 이름의 서비스 아이디어를 구상했다.

Catnap은 사용자가 콘텐츠를 고르기 어려운 순간에, 직접 긴 검색어를 입력하지 않아도 현재 의도를 표현할 수 있게 하는 서비스이다. 사용자는 먼저 `밥친구 모드`, `잠들기 전 모드`, `짧게 웃고 싶음 모드`처럼 콘텐츠를 보려는 상황을 고른다. 이후 카드 선택을 통해 선호 장르, 분위기, 길이, 피하고 싶은 요소를 입력한다. 이때 카드 UI는 사용자가 막연한 요구를 쉽게 표현하도록 돕는 입력 방식이고, 기술적으로 중요한 지점은 이 선택값을 검색 가능한 사용자 요구 프로필로 변환하는 과정이다.

```markdown
사용자 선택값
→ 사용자 요구 프로필 생성
→ Embedding
→ 콘텐츠 설명문 벡터와 비교
→ Vector Search 기반 Top-K 후보 반환
→ 조건 기반 재정렬
→ 최종 추천 결과
```

예를 들어 사용자가 `밥친구 모드`, `먹방`, `짧은 영상`, `가벼운 분위기`를 선호하고 `공포`, `긴 다큐`를 회피 조건으로 선택했다면, 시스템은 이를 다음과 같은 사용자 요구 프로필로 바꿀 수 있다.

```markdown
"사용자는 밥을 먹으면서 부담 없이 볼 수 있는 짧고 가벼운 콘텐츠를 원한다.
먹방이나 예능형 콘텐츠에 긍정적이며, 공포나 긴 다큐처럼 몰입 부담이 큰 콘텐츠는 피하고 싶어 한다."
```

이 사용자 요구 프로필은 Embedding을 통해 벡터화되고, 영상 제목·장르·길이·분위기·시청 상황·키워드 등을 포함한 콘텐츠 설명문 벡터와 비교된다. Vector Search는 의미적으로 가까운 Top-K 후보를 반환하고, 조건 기반 재정렬은 선호 조건과 회피 조건을 반영해 최종 순위를 조정한다.

```markdown
1차 검색:
사용자 요구 프로필 벡터
→ 콘텐츠 설명문 벡터들과 유사도 비교
→ Top-K 후보 반환

조건 기반 재정렬:
final_score = vector_similarity
            + positive_match_score
            - negative_match_penalty
            - duration_penalty
```

Catnap은 워드클라우드 데모에서 확인한 의미 기반 검색 원리를 단어 검색에만 머무르게 하지 않고, 사용자의 상황과 취향을 반영한 콘텐츠 추천 흐름으로 확장하는 방향을 보여준다. 데모가 Vector Search의 작동 원리를 시각적으로 보여준다면, Catnap은 그 기술이 실제 사용자 경험 안에서 어떤 형태로 활용될 수 있는지를 보여주는 아이디어 기획이다.

# 6. 정리

이번 콘텐츠는 사용자가 콘텐츠 플랫폼에서 겪는 선택 피로에서 출발했다. 콘텐츠는 많아졌지만, 사용자는 자신이 지금 원하는 콘텐츠를 정확한 검색어로 표현하기 어렵다. 특히 밥을 먹을 때, 잠들기 전, 쉬는 시간처럼 가볍게 콘텐츠를 보고 싶은 순간에는 “무언가 흥미로운 것”이라는 막연한 요구만 있고, 이를 명확한 검색어로 바꾸기는 쉽지 않다.

이를 해결하기 위한 기술 흐름으로 사용자 요구 프로필 생성, 콘텐츠 설명문 구성, Embedding, Vector Search, Top-K Retrieval, 조건 기반 재정렬을 다루었다. 청중 참여형 워드클라우드 데모는 이 중 Embedding, Vector Search, Top-K Retrieval의 원리를 실제 화면에서 보여준다. 청중이 입력한 단어는 검색 가능한 후보군이 되고, 발표자가 검색어를 입력하면 의미적으로 가까운 Top-K 단어가 워드클라우드에서 강조된다.

Catnap은 이 기술 흐름을 실제 콘텐츠 추천 경험으로 확장한 아이디어 시안이다. 사용자는 지금 모드와 카드 선택을 통해 막연한 현재 의도를 표현하고, 시스템은 이를 사용자 요구 프로필로 변환한 뒤 콘텐츠 설명문과 벡터 공간에서 비교한다. 결국 핵심은 사용자의 막연한 요구를 검색 가능한 의미 정보로 바꾸고, 이를 Vector Search 기반 추천 흐름으로 연결하는 데 있다.
