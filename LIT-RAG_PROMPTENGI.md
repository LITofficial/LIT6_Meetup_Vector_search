
# LLM 답변 품질을 높이기 위한 맥락 설계: 프롬프트 엔지니어링과 RAG

# 1. 서론

## 1.1 도입: 일상 속 LLM

최근 몇 년 사이 LLM은 빠르게 발전했고, 이제 많은 사람들이 ChatGPT, Gemini, Claude 같은 서비스를 일상적으로 사용하고 있습니다.

그 규모도 매우 큽니다. OpenAI는 2026년 2월 기준 ChatGPT의 **주간 활성 사용자 수가 9억 명 이상**이라고 밝혔고, Google 역시 2026년 I/O에서 Gemini 앱의 **월간 활성 사용자 수가 9억 명을 넘었다**고 발표했습니다.

물론 두 수치는 기준이 다릅니다. ChatGPT는 **주간 활성 사용자**, Gemini는 **월간 활성 사용자** 기준이기 때문에 단순히 합쳐서 비교할 수는 없습니다. 하지만 분명한 점은, LLM이 더 이상 일부 개발자나 연구자만 사용하는 기술이 아니라는 것입니다. 이제 LLM은 글쓰기, 검색, 요약, 코딩, 학습 등 일상에서 널리 쓰이는 도구가 되었습니다.

아마 이 발표를 듣는 분들 중에도 LLM을 한 번쯤 사용해본 분들이 많을 것입니다. 그런데 사용하다 보면 항상 만족스러운 답변만 나오는 것은 아닙니다.

예를 들어 이런 경험이 있을 수 있습니다.

- 너무 일반적인 답변이 나온다.
- 내가 원한 방향과 다른 답변이 나온다.
- 근거가 부족한 답변이 나온다.
- 바로 쓰기 어려운 답변이 나온다.

저는 이런 경험을 할 때마다 한 가지 의문이 들었습니다.

> **왜 같은 AI를 쓰는데도 답변 품질이 달라질까?**

이번 발표는 이 질문에서 출발합니다.

## 1.2 문제 제기: 답변 품질은 입력에 영향을 받는다

LLM의 답변 품질이 달라지는 이유를 이해하려면, 먼저 LLM이 어떤 방식으로 답변을 생성하는지 간단히 살펴볼 필요가 있습니다.

LLM은 마법처럼 사용자의 머릿속 의도를 읽는 도구가 아닙니다.  
사용자가 입력한 문장과 그 안에 담긴 맥락을 바탕으로 답변을 생성하는 모델입니다.

따라서 답변 품질은 모델 자체의 성능뿐만 아니라, 사용자가 어떤 입력을 제공하느냐에도 영향을 받습니다.

같은 모델을 사용하더라도 질문이 모호하면 답변도 일반적이거나 애매해질 수 있습니다. 반대로 목적, 조건, 필요한 배경 정보가 명확하게 주어지면 더 원하는 결과에 가까운 답변을 얻을 수 있습니다.

즉, 좋은 답변을 얻기 위해서는 먼저 **LLM이 입력을 어떻게 처리하고, 어떤 방식으로 답변을 만들어내는지** 이해할 필요가 있습니다.

그래서 다음으로는 LLM의 동작 원리를 아주 간단히 살펴보겠습니다.

# 2. 본론
## 2.1 LLM 원리 최소 설명

LLM은 입력된 문맥을 바탕으로, 다음에 올 가능성이 높은 단어 또는 토큰을 예측하며 답변을 생성합니다.

여기서 토큰은 쉽게 말해 단어 또는 단어의 조각이라고 생각하면 됩니다.

예를 들어 설명용으로 아주 단순화해서 보면,

> **The cat is ...**

라는 입력이 들어왔을 때, LLM은 앞의 문맥을 참고하여 다음에 올 수 있는 후보들의 가능성을 계산합니다.

| 후보 | 확률 |
|---|---:|
| a | 76% |
| cute | 14% |
| black | 10% |

물론 실제 모델은 이보다 훨씬 복잡하게 동작하지만, 핵심 아이디어는 비슷합니다.  
LLM은 앞에 주어진 내용을 참고해 다음에 올 가능성이 높은 표현을 예측하고, 이 과정을 반복하면서 문장을 생성합니다.

즉, LLM의 답변에 영향을 주는 중요한 요소는 다음과 같습니다.

- 앞에 어떤 문장이 주어졌는지
- 어떤 조건이 주어졌는지
- 어떤 참고 자료와 배경 정보가 함께 제공되었는지

결국 LLM은 아무 정보 없이 사용자의 의도를 알아내는 것이 아니라, **입력으로 주어진 맥락을 바탕으로 답변을 만들어내는 모델**입니다.

내용이 조금 복잡해 보일 수 있지만, 간단한 비유로 생각하면 이해하기 쉽습니다.

> LLM은 시험을 치는 학생과 비슷합니다.  
> 문제만 주면 일반적인 답을 쓰지만, 채점 기준, 참고자료, 답안 형식을 함께 주면 더 원하는 답에 가까운 답을 쓸 수 있습니다.

따라서 좋은 답변을 얻으려면, 단순히 질문만 던지는 것이 아니라 LLM이 답변을 잘 만들 수 있도록 입력을 설계해야 합니다.

그렇다면 우리는 LLM에게 어떤 방식으로 입력을 주어야 할까요?

이 질문에서 이어지는 개념이 바로 **프롬프트 엔지니어링**입니다.


## 2.2 프롬프트 엔지니어링

프롬프트 엔지니어링은 LLM이 사용자의 의도에 더 가까운 답변을 생성하도록 **입력을 설계하는 과정**입니다.

앞서 살펴본 것처럼, LLM은 입력된 문장과 맥락을 바탕으로 답변을 생성합니다.  
따라서 원하는 답변을 얻기 위해서는 단순히 질문만 던지는 것이 아니라, LLM이 참고할 수 있는 조건과 맥락을 명확하게 제공해야 합니다.

좋은 프롬프트에는 보통 다음과 같은 요소들이 포함됩니다.

| 요소 | 설명 |
|---|---|
| 역할 | 어떤 관점에서 답할지 |
| 목표 | 무엇을 얻고 싶은지 |
| 조건 | 지켜야 할 제한사항 |
| 맥락 | 참고해야 할 배경 정보 |
| 출력 형식 | 표, 목록, 발표문 등 |
| 예시 | 원하는 답변 스타일 |

예를 들어, 다음 두 프롬프트를 비교해볼 수 있습니다.

```text
나쁜 예시:
RAG 설명해줘.

좋은 예시:
비전공자도 이해할 수 있도록 RAG를 설명해줘.
프롬프트 엔지니어링과의 관계를 포함하고,
발표 슬라이드에 넣을 수 있게 3문장으로 정리해줘.
```

첫 번째 프롬프트는 질문이 너무 짧고 목적이 명확하지 않습니다.  
반면 두 번째 프롬프트는 대상, 설명 방식, 포함할 내용, 출력 형식이 더 구체적으로 제시되어 있습니다.

결국 프롬프트 엔지니어링은 AI에게 질문을 예쁘게 쓰는 기술이 아닙니다.  
**원하는 답변이 나오도록 입력 조건과 맥락을 설계하는 기술**입니다.


## 2.3 핵심 전환: 프롬프트 엔지니어링과 맥락(실제 답변 차이 이미지 넣으면 좋을듯)

앞서 살펴본 것처럼, 좋은 프롬프트에서 특히 중요한 요소는 **맥락**입니다.

질문만 던지면 LLM은 일반적이거나 애매한 답변을 생성할 가능성이 높습니다.  

반대로 목적, 대상, 자료, 조건과 같은 맥락을 함께 제공하면 답변은 훨씬 구체적이고 유용해집니다.
		
예를 들어 다음과 같이 질문했다고 가정해보겠습니다.

```text
발표 주제 추천해줘.
```

이 질문만으로도 답변은 나오겠지만, 발표 대상이나 주제 범위, 발표 시간, 데모 필요 여부를 알 수 없기 때문에 일반적인 추천에 그칠 가능성이 큽니다.

하지만 아래와 같이 맥락을 함께 제공하면 결과가 달라집니다.

```text
대상: 비전공자 포함 대학생
주제 범위: Microsoft Learn 기반 AI 기술
조건: 데모 포함
발표 시간: 10분
관심 주제: 프롬프트 엔지니어링과 RAG
```

이 경우 LLM은 단순히 발표 주제를 아무거나 추천하는 것이 아니라, 주어진 조건에 맞춰 더 구체적이고 목적에 맞는 답변을 생성할 수 있습니다.
  
즉, 프롬프트 엔지니어링의 핵심은 단순히 질문을 길게 쓰는 것이 아니라, **LLM이 답변을 만들 때 참고할 수 있는 적절한 맥락을 제공하는 것**입니다.

> 좋은 답변은 좋은 질문에서 시작하지만,  
> 더 좋은 답변은 좋은 맥락에서 완성됩니다.


## 2.4 RAG

프롬프트 엔지니어링에서 살펴봤듯이, LLM에게 적절한 **맥락**을 제공하는 것은 좋은 답변을 얻는 데 매우 중요합니다.

맥락은 텍스트로 직접 입력할 수도 있고, **파일이나 문서 형태로 제공**할 수도 있습니다.  
예를 들어 강의자료, 발표자료, 회사 매뉴얼, 논문 PDF 같은 자료를 함께 제공하면 LLM은 그 내용을 참고하여 더 구체적인 답변을 만들 수 있습니다.

하지만 여기서 한 가지 문제가 생깁니다.

질문을 할 때마다 사용자가 직접 관련 파일을 찾고, 필요한 내용을 골라서 넣어야 한다면 어떨까요? 

자료가 한두 개라면 괜찮겠지만, 문서가 많아질수록 어떤 자료를 넣어야 할지 판단하기도 어렵고, 매번 반복하기도 번거롭습니다.

이럴 때 사용할 수 있는 방식이 바로 **RAG**입니다.

RAG는 **Retrieval-Augmented Generation**의 약자로, 한국어로는 보통 **검색 증강 생성**이라고 설명할 수 있습니다.

쉽게 말하면, RAG는 여러 자료를 미리 넣어두고, 사용자의 질문과 관련 있는 자료를 검색한 뒤, 그 내용을 프롬프트의 맥락으로 추가해서 LLM이 답변하게 하는 방식입니다.

RAG의 작동 흐름은 이와 같습니다.

```
자료 업로드
→ 문서 조각화
→ 검색 가능한 형태로 저장
→ 사용자 질문
→ 관련 자료 검색
→ 검색된 내용을 프롬프트에 추가
→ LLM 답변 생성
```

쉽게 비유하면

> RAG는 AI에게 자료실과 사서를 붙여주는 방식입니다.  
> 질문이 들어오면 사서가 관련 자료를 먼저 찾아주고, AI는 그 자료를 바탕으로 답변합니다.

즉, RAG는 LLM 자체를 새로 학습시키는 것이 아니라, 답변할 때 필요한 자료를 찾아서 **맥락으로 제공하는 방식**입니다.

프롬프트 엔지니어링이 사용자가 직접 좋은 입력을 설계하는 과정이라면,  
RAG는 그중에서도 특히 중요한 **참고 자료와 맥락을 자동으로 찾아 붙여주는 방식**이라고 볼 수 있죠.

요약하자면,

> **RAG는 좋은 프롬프트에 필요한 맥락을 자동으로 찾아 붙여주는 기술입니다.**

(나중에 굳이 RAG를 써야하는 이유에 대해서도 언급해야 할듯.)

# 3. 결론

## 3.1 활용 가능성

오늘은 프롬프트 엔지니어링과 RAG를 통해 LLM에게 더 좋은 답변을 얻는 방법을 살펴보았습니다.

그런데 여기서 이런 의문이 들 수 있습니다.

> 그냥 ChatGPT, Gemini, Claude에 파일을 넣고 질문하면 되는 것 아닌가?

개인적으로 간단히 사용할 때는 충분히 좋은 방법입니다.  
하지만 자료가 많아지고, 같은 자료를 반복적으로 사용해야 하거나, 여러 사람이 함께 활용해야 하는 상황이라면 이야기가 달라집니다.

일반적인 LLM 서비스에 파일을 직접 넣어 사용하는 방식은 편리하지만, 자료 관리, 검색, 업데이트, 반복 사용 측면에서 한계와 귀찮음이 생길 수 있습니다.

반면 RAG는 여러 자료를 미리 정리해두고, 질문과 관련된 내용을 찾아 맥락으로 제공할 수 있기 때문에 더 체계적으로 활용할 수 있습니다.

예를 들어 RAG는 다음과 같은 상황에서 유용하게 사용할 수 있습니다.

- 수업 자료 Q&A
- 회사 매뉴얼 챗봇
- 논문 읽기 보조
- 동아리 문서 검색
- 고객 문의 자동화

즉, RAG를 통해 좋은 맥락을 더 정확히 제공함으로써 더 좋고 원하는 답변을 얻을 수 있습니다.

## 3.2 정리

이번 발표의 핵심은 다음과 같습니다.

```text
LLM은 입력된 맥락을 바탕으로 답변을 생성한다.
그래서 좋은 답변을 얻기 위해서는 좋은 질문뿐 아니라 좋은 맥락도 필요하다.
프롬프트 엔지니어링은 그 입력을 잘 설계하는 방법이고,
RAG는 필요한 맥락을 자료에서 찾아 자동으로 추가하는 방식이다.
```

결국 LLM을 잘 활용한다는 것은 단순히 AI에게 질문을 던지는 것이 아닙니다.  
AI가 좋은 답변을 만들 수 있도록 **목적, 조건, 형식, 참고 자료와 같은 맥락을 잘 제공하는 것**입니다.

이제 중요한 것은 단순히 AI를 쓰는가가 아니라, **AI를 어떻게 더 잘 활용할 수 있는가**입니다.

앞으로 더 정확하고 원하는 답변을 얻고 싶거나, 참고해야 할 자료가 많아 관리가 어려운 상황이라면 RAG를 활용하는 것을 고려해보는 게 어떤가요?
